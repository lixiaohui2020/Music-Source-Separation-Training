## 1. 背景与目标

- 公司已有自研 NPU，面向图像负载优化（2D MAC 阵列、INT8、大批量推理），无音频专门适配。
- 算法团队已梳理 10 个应用场景、36 个模型的算子需求，计划与 NPU 数字设计团队共同定义 Audio Core。
- 本报告回答三个问题：
  1. **要不要做**（必要性论证，含诚实的成本侧）；
  2. **怎么做**（方案选型 → 架构 → 规格 → 软件栈 → 集成）；
  3. **怎么推进**（实测先行的立项流程、分阶段交付、协作接口）。

**定位一句话：Audio Core 是"AI 音频推理加速器 + 前端特征引擎"，不是传统 Audio DSP。** 不做 AEC/AGC/DRC/LMS 等传统信号链，只做 AI 推理路径上 NPU 做不了或做不好的部分。

## 2. 需求分析：场景、模型与算子

### 2.1 场景分级

| 梯队 | 场景 | 端侧位置 | 核心模型结构 | 关键算子 |
|---|---|---|---|---|
| **P0** | 降噪 | 拾音端 | GRU + LSTM（DCCRN / GTCRN） | STFT/ISTFT、Conv1D、RNN gate |
| **P0** | 人声分离 | 播放端 | Transformer / LSTM | STFT/ISTFT、Attention、LSTM |
| **P0** | 唤醒 / VAD | 拾音端，**always-on** | Conv1D KWS | MFCC、Conv1D |
| **P1** | 语音识别 | 拾音端 | Transformer（Conformer/Paraformer） | MFCC、Attention、MatMul |
| **P1** | TTS | 播放端 | Transformer | Attention、MatMul、ISTFT |
| **P1** | 声纹识别 | 拾音端 | Conv1D（ECAPA-TDNN） | MFCC、Conv1D |
| **P2** | 说话人日志 | — | LSTM | STFT、LSTM |
| **P2** | 音频标记 | 拾音端 | Transformer（AST） | STFT/MFCC、Attention |

### 2.2 公共基础算子（AFE 层）

所有 36 个模型共同依赖四个前端算子，且其数学定义长期稳定（不随模型换代而变化）：

| 算子 | 功能 | 计算瓶颈 | 存储瓶颈 |
|---|---|---|---|
| **STFT** | 时频变换 | FFT 蝶形运算 | Window ROM、Twiddle ROM |
| **ISTFT** | 频时逆变换 | IFFT 蝶形 + OLA | 合成窗 ROM + OLA Buffer |
| **MFCC** | 梅尔倒谱特征 | FFT 蝶形 | Mel 滤波器组（可稀疏化，见 §6.4） |
| **重采样** | 采样率转换 | 多相 FIR | 滤波器系数 ROM |

STFT 计算量示例（人声分离最重参数）：N=4096, hop=1024, fs=44.1kHz → 每秒 43.07 帧 × 每帧约 77.8K 次运算 ≈ **3.35M ops/s**。绝对算力不大，问题在执行效率与调度粒度（见 §3）。

### 2.3 关键应用参数组合（需算法侧最终确认）

| 应用 | fs | n_fft | hop | win | center | pad | normalized |
|---|---|---|---|---|---|---|---|
| AW 降噪 | 16k | **320（原表 16000 疑为笔误，待确认）** | 160 | 320 | False | — | False |
| 降噪 DCCRN | 16k | 512 | 100 | 400 | True | reflect | False |
| 降噪 GTCRN | 16k | 512 | 256 | 512 | True | reflect | False |
| AW 人声分离 | 44.1k | 4096 | 1024 | 4096 | True | constant | True |
| 人声分离 2 | 44.1k | 2048 | 512 | 2048 | — | — | — |
| Whisper (MFCC) | 16k | 400 | 160 | 400 | — | — | n_mels=128/80, hann |
| Conformer/Paraformer | 16k | 400 | 160 | 400 | — | — | n_mels=80, hamming |
| KWS | 16k | 400 | 160 | 400 | — | — | n_mels=80, povey |

> **行动项（算法团队）**：冻结此表。它直接决定：① 是否需要 320/400 点 mixed-radix FFT；② COLA 约束下 ISTFT 归一化能否简化为常数缩放（§6.2）；③ Window ROM 的种类（hann/hamming/povey）。

### 2.4 模型规模实况（修正前版错误）

| 模型 | 参数量（公开数据） | 每帧计算量 | 跨帧状态 |
|---|---|---|---|
| Conv1D KWS（唤醒） | 50~200K | 小 | 无状态 |
| **GTCRN** | **~23.7K（前版误写 1~3M）** | ~33 MMACs/s | GRU hidden ~0.5–1 KB |
| **DCCRN** | **~3.7M** | 每帧 ~数 M MAC | LSTM hidden ~1–2 KB |
| ECAPA-TDNN（声纹） | 5~20M | 非实时批处理为主 | 无状态 |

两个修正带来的结论变化：

- **GTCRN 极小（24K 参数）**：在图像 NPU 上跑，任务下发/调度开销占比会非常高，专用引擎的相对收益反而更大。这是有利论据，前版的错误数字掩盖了它。
- **DCCRN 权重（INT16 约 7.4MB）放不进任何合理规模的 Audio Core SRAM**，真实瓶颈是权重带宽而非 hidden state（详细算账见 §7）。前版"hidden state 搬运是访存大户"的判断需要修正。

## 3. 现有图像 NPU 的错配分析

> 措辞原则：区分**架构事实**（可直接论证）与**量化估计**（必须以本公司 NPU 实测为准）。下表中带 ★ 的数字是估计值，立项前须用 §14 Step 1 的实测替换。

### 3.1 六维错配

| 维度 | 图像 NN（NPU 设计假设） | AI 音频 NN（实际负载） | 后果 |
|---|---|---|---|
| 卷积形态 | Conv2D k=3×3，大批量 | Conv1D k=3/5/11，长序列、小 channel | 2D 阵列利用率低（★估 16~20%，待实测） |
| 执行模式 | 整图单次推理 | 逐帧流式、causal（不可看未来帧） | causal padding / streaming 需软件绕行 |
| 状态 | 无状态 | RNN hidden / KV cache 跨帧保持 | 帧间状态往返总线，破坏流水 |
| 调度粒度 | ~30ms 级任务 | 6~10ms 一帧（hop=100~160 samples） | 任务下发/中断开销占比过高 |
| 精度 | INT8 足够 | 前端需 INT16/Q15，累加 32~40bit | INT8 前端量化噪声可闻（底噪/振铃） |
| 前后处理 | 无（像素进出） | STFT/MFCC/重采样必经，占总算力 10~30% | NPU 无原生 FFT 数据通路 |

### 3.2 对"NPU 做不了 STFT"的严谨修正

STFT **可以**在 NPU 上做：把加窗 DFT 展开为矩阵乘（Whisper 的很多 NPU 部署即如此，n_fft=400 时 DFT 矩阵仅 400×402，可行）。准确的结论是：

1. matmul 实现是 O(N²)，FFT 是 O(N·logN)——**N=4096 时代价差约 340×**，人声分离场景不可接受；
2. 逐帧（6~10ms）调用 NPU 做小矩阵乘，调度开销淹没计算本身；
3. INT8 NPU 通路的前端精度不满足音质要求。

即：**小点数"能做但不划算"，大点数"接近不可行"**。这一表述在与 NPU 团队论证时更站得住。

### 3.3 Always-on 唤醒的功耗鸿沟（最硬的一条）

- VAD/唤醒要求 always-on，功耗预算 **<1mW**（可穿戴/TWS 场景）；
- 图像 NPU 的待机+启动功耗在几十 mW 量级（★具体值待实测），与预算差 1~2 个数量级；
- **这不是优化问题，是电源域架构问题**：必须有独立低功耗域的小算力单元。此条不依赖任何利用率估算，是必要性论证中最不可辩驳的一条。

**核心结论：矛盾不是算力总量，而是执行模型（流式/状态/粒度/精度/功耗域）的系统性错配。**

## 4. 方案选型：四条技术路线对比

> 前版直接跳到"自研固定功能引擎"，跳过了选型。这是立项评审必然被问到的问题，应显式回答。

| 路线 | 内容 | 优点 | 缺点 | 结论 |
|---|---|---|---|---|
| **A. 扩展现有 NPU** | 加 Conv1D 模式、native LSTM、INT16 通路 | 复用编译器/工具链；团队熟悉 | 解决不了 always-on 功耗域问题；对 NPU 侵入式改动，验证归零 | 不推荐单独采用；部分思路可并入 NPU 下一代 |
| **B. 外购音频 DSP IP** | Cadence HiFi 5/1s、CEVA、ARC EM 等 | 工具链成熟、生态有大量参考实现、风险低 | License 成本；能效低于专用电路；无差异化；AI 算子仍需 NN 扩展 | 作为兜底与对标基线；若自研人力不足则是务实选择 |
| **C. 自研全固定功能** | AFE + Conv1D + RNN 全硬化（前版方案） | PPA 最优 | **灵活性风险最大**：网络结构 2~3 年换代（流式 attention、神经 codec、learned 前端），硬化 RNN 引擎可能变暗硅；软件栈全自研 | 不推荐全硬化 |
| **D. 混合（推荐）** | **AFE 硬加速器（FFT/FIR/mel/LUT 硬化）+ 小可编程核（RISC-V+向量扩展或微码引擎）执行 NN 层** | 硬化部分只锁定数学上不变的算子，拿走大头能效收益；可编程部分兜底模型演进 | 面积略大于 C；仍需自研软件栈（但范围可控） | **推荐** |

**D 路线的硬化/可编程边界原则**：
- **硬化**：FFT/IFFT（含 mixed-radix 320/400）、多相 FIR 重采样、mel 三角滤波、窗函数乘加、非线性激活 LUT（sigmoid/tanh/exp/log）、OLA 缓冲。这些数学定义十年不变。
- **可编程/可微码**：Conv1D、LSTM/GRU gate 组合、逐层调度、量化/反量化。MAC 阵列可以是硬件，但**层的编排必须软件可控**，以适应模型换代。

## 6. AFE Engine 关键设计规格

### 6.1 FFT / IFFT Core

| 设计项 | 规格 | 依据 |
|---|---|---|
| 点数（硬支持） | 256 / 512 / 1024 / 2048 / 4096，radix-2 DIT | 覆盖 §2.3 参数表 |
| 非 2 的幂 | **320 = 2⁶×5、400 = 2⁴×5²**：mixed-radix（radix-2 + radix-5）可实现，复杂度可控；其余任意点数 SW fallback | Whisper/Conformer 的 400、AW 的 320 均为高频参数，建议硬支持而非 optional |
| FFT/IFFT 共享 | mode bit 切换，Twiddle 取共轭；输入 bit-reverse（2 的幂）/ 定制重排（320/400） | 复用蝶形阵列 |
| 位宽 | 输入 Q15；Twiddle Q15；累加器 **40bit**；输出 round-to-nearest-even 截断 Q15 | N=4096 时幅度最大增长 4096×（12bit），须防溢出 |
| 缩放 | IFFT 末级除 N（2 的幂用移位；320/400 用常数乘） | — |
| onesided | 硬件 wrapper 完成 Hermitian 共轭对称扩展（N/2+1 → N） | 省软件搬运 |

### 6.2 ISTFT / OLA（含前版忽略的除法问题）

流程：`full spectrum → IFFT → 合成窗 → OLA 累加 → 包络归一化 → center 裁剪 → 输出`。

**归一化除法的硬件处理（关键决策）**：
- OLA 包络归一化 `y[n] / Σ_t w²[n − tH]` 逐点除法代价高；
- **方案**：spec 中声明仅支持满足 **COLA 条件**的 (window, hop) 组合 → 包络为常数 → 除法退化为一次常数缩放（乘法）；
- 非 COLA 组合走倒数 LUT + 乘法的慢速通路，或软件 fallback；
- 需算法侧确认 §2.3 全部组合的 COLA 性（hann + hop=N/4、N/2 均满足）。

其他：OLA Buffer 用环形寻址 + 硬件偏移计数器（hop=100/160 非 2 的幂，不能依赖软件取模）；center=True 时输出裁掉两端 N/2。

### 6.3 MFCC

- 流程：预加重 → 分帧加窗 → FFT（复用 FFT Core）→ Mel 滤波 → log；**DCT 不做硬化**（语音识别/唤醒主流模型均不用 DCT，需要时软件做）。
- log 用硬件 LUT（与激活 LUT 共享架构）。
- 窗 ROM：hann / hamming / povey（hann^0.85）三套 Q15 系数。

### 6.4 Mel 滤波器组稀疏存储

- 不存稠密矩阵（80 mel × 201 bin = 16,080 条目）；
- 存稀疏表：每个滤波器的起止 bin + 上升/下降斜率（或全部非零权值+地址），80 mel × 平均 ~6 个非零 bin ≈ **~480 条目**，存储降约 30×。

### 6.5 重采样（多相 FIR）

| 优先级 | 转换 | 备注 |
|---|---|---|
| P0 | 48k→16k、48k→8k、16k→48k | 整数比，多相分解，计算量降 M/L 倍 |
| P1 | 32k↔16k、24k↔16k | 整数比 |
| P2 | **44.1k↔48k** | 分数比 **147/160**，多相相位数 160，系数存储显著增大，需单独评估 ROM 预算 |
| P3 | 任意异步 ASRC | 需要 Farrow 结构/多项式插值，建议二期 |

设计参考（48k→16k）：通带 0~7.2kHz、阻带 8~24kHz、Astop=80dB → Kaiser 估计 **~601 taps**；多相分解后每输出样本 ~200 MAC。

## 7. 关键瓶颈：权重带宽与存储预算

> 前版判断"hidden state 访存是大户"——hidden state 只有 1~2KB，不是主要矛盾。**真正的账在权重。**

### 7.1 DCCRN 权重带宽算账

- 参数量 ~3.7M，INT16 权重 ≈ **7.4MB**；
- hop=100 @16kHz → 每秒 160 帧，每帧 6.25ms 内须过一遍全部权重；
- 若权重放 DRAM：**持续带宽 ≈ 7.4MB × 160 ≈ 1.18GB/s**，对低功耗音频子系统不可接受；
- 可选对策（按优先级）：
  1. **权重压缩**：4bit 权重 + 硬件解压 → 1.85MB，配 2MB 级 SRAM 可全驻留（面积代价需评估）；
  2. **模型侧解决**：主推 GTCRN 路线（24K 参数，SRAM 轻松全驻留），DCCRN 仅作为高端配置；
  3. **分层驻留**：LSTM bottleneck 权重驻留 SRAM，Conv encoder/decoder 权重流式或留 NPU；
- **此账直接决定 SRAM 容量档位（256KB / 512KB / 2MB）与是否需要权重解压单元，必须在架构冻结前与算法团队一起敲定模型路线。**

### 7.2 SRAM 预算（按模型路线分档）

| 配置 | 覆盖模型 | SRAM 建议 |
|---|---|---|
| 基础档 | KWS + GTCRN + AFE 全功能 | 256~512KB（KWS ~50KB + GTCRN 权重/激活 <100KB + 帧 buffer/OLA） |
| 增强档 | + DCCRN（4bit 压缩全驻留） | ~2MB |

### 7.3 Hidden state 处理（仍要做，但定位是"顺手优化"）

- LSTM/GRU hidden + cell state 共 0.5~2KB，锁 **NN 执行单元本地寄存器组**，不走总线；
- 收益是消除每帧 5~10μs 的往返延迟，保障流水，而非解决带宽问题。

## 8. 混合精度策略

| 层次 | 精度 | 原因 |
|---|---|---|
| STFT / ISTFT / 重采样 | Q15（INT16×INT16 → INT32/40 acc） | 前端精度 = 音质，量化噪声直接可闻 |
| Conv1D（唤醒/声纹） | INT8 weight / INT16 act | 该类模型对量化鲁棒 |
| RNN gate（降噪） | INT16 weight / INT32 acc | RNN 逐帧误差累积比 CNN 严重，INT8 会引入可闻噪声 |
| DNN head（唤醒后端） | INT8 | 量化友好 |
| 与 NPU 交界处 | 明确 requant 归属（见 §15） | Q15 ↔ INT8 的转换点必须写进接口协议 |

> 行动项（算法团队）：对每条精度假设做**定点仿真验证**（Q15 STFT + INT16 RNN 链路），在 RTL 之前给出各场景定点 vs FP32 的实测劣化量，替换上表中的经验判断。

## 10. 系统级集成：电源域、唤醒链与并发

### 10.1 电源域与唤醒链（"30×功耗收益"能否兑现的前提）

```
[PDM mic] → [Audio Core always-on 域]
              ├─ VAD（μW 级，模拟或极简数字）
              ├─ KWS Conv1D（<1mW，检测到活动话语时运行）
              └─ 命中唤醒词 → 中断 PMU → 上电 NPU/CPU 域
```

- Audio Core 必须位于**独立低功耗电源域**，带自己的时钟源（32k/慢时钟 + 按需 PLL）与保持 SRAM；
- 电源域划分、retention 策略、唤醒延迟预算（目标 1~3ms）需在 SoC 架构层面与设计团队共同定义——**这比 MAC 阵列本身更决定功耗收益**。

### 10.2 音频数据通路

- PDM/I2S/TDM 接口直入 Audio Core 帧 buffer（不经 CPU/DDR），多 mic TDM 支持波束前端的数据摆放；
- Audio Core ↔ NPU：共享 SRAM 或 AXI + 帧中断/ready-valid 握手（接口冻结见 §15）。

### 10.3 并发场景（对管理层最有力的产品论据）

**通话降噪（Audio Core 常驻）+ 本地 ASR/LLM（NPU）同时运行**——单 NPU 分时无法保证音频硬实时（6~10ms 帧节拍），双引擎架构是本地语音交互产品（会议记录、实时翻译、车载助手）的使能条件，不只是省电。

### 10.4 调度归属

- 帧级流水由 **Audio Core 内置 sequencer** 编排（AFE→NN→输出全帧自治），不要用 CPU 每 10ms 中断驱动（CPU 负担 + 抖动）；
- CPU 只做场景级配置与异常处理。

## 11. 验证策略与验收标准

### 11.1 功能验证（数字团队 + 算法团队）

- **唯一 sign-off 标准：bit-exact 对拍 golden vectors**（算法团队用锁定版参考实现生成，覆盖 §2.3 全部参数组合 + 边界条件：溢出、极短输入、非 COLA 组合 fallback 等）；
- CI 全量回归。

### 11.2 端到端质量验收（算法团队定义，立项时冻结阈值）

| 场景 | 指标 | 建议阈值（定点 vs FP32 参考） |
|---|---|---|
| 降噪 | PESQ / STOI | PESQ 劣化 ≤ 0.05 |
| 唤醒 | 唤醒率 @ 固定误唤醒 | 劣化 ≤ 0.5% |
| 人声分离 | SDR / SI-SNR | 劣化 ≤ 0.3dB |
| 语音识别 | WER | 相对上升 ≤ 1% |
| 声纹 | EER | 劣化 ≤ 0.2% |
| 说话人日志 | DER | 劣化 ≤ 1% |

> 该表既是 DV 验收标准，也是 §8 混合精度选型争议的最终裁决依据。

### 11.3 性能/功耗验收

| 项 | 目标值（★立项后以实测校准） |
|---|---|
| 唤醒链路功耗（always-on 域） | < 1mW（VAD < 100μW） |
| 唤醒响应延迟 | 1~3ms |
| GTCRN 端到端帧延迟 | < 10ms（含 STFT/ISTFT） |
| AFE 吞吐 | ≥ 3.35M ops/s（覆盖 44.1k/4096 人声分离） |
| Conv1D 阵列利用率 | > 85% |

## 12. 必要性与收益（含成本）

### 12.1 必要性：三条硬理由（按不可辩驳程度排序）

1. **Always-on 功耗物理不可行**：图像 NPU 待机/启动功耗与 VAD/唤醒 <1mW 预算差 1~2 个数量级，是电源域架构问题而非优化问题。**必须**有独立低功耗域的小算力单元。
2. **流式执行模型系统性错配**：6~10ms 帧粒度 + causal streaming + 跨帧状态，与图像 NPU 的"整图、无状态、大批量"设计假设根本冲突；GTCRN 这类 24K 参数模型在 NPU 上调度开销占比极高。
3. **前端音质的精度刚性**：STFT/ISTFT 需 Q15 + 32~40bit 累加，INT8 通路的量化噪声可闻，前端必须有独立高精度定点通路。

### 12.2 收益（如实标注：目标值，待 §14 Step 1 实测校准）

| 维度 | 目标 |
|---|---|
| 唤醒/VAD 功耗 | 进入 sub-mW，可穿戴/TWS 续航直接受益（★目标 30× 级改善） |
| P0 降噪能效 | ★10× 量级（以 DCCRN/GTCRN 实测为准） |
| **NPU 释放与并发** | 音频常驻任务不占 NPU → 本地 LLM/ASR 可与降噪同时运行（**产品差异化，非仅省电**） |
| 前端经济性 | FFT 类负载专用电路 vs NPU matmul 展开，能效差 ★10~25× |
| 行业验证 | "音频单元 + NPU"是成熟标准架构（§13），非激进方案 |

### 12.3 成本（诚实列出，立项文档必须包含）

- 面积：0.3~0.8mm² @28nm 量级（按 §7 SRAM 档位浮动）；
- 人力：RTL + DV + **全套软件工具链**（编译器切分、量化工具、仿真器——软件占总工程量一半以上）；
- 风险：模型演进导致硬化逻辑闲置（已通过 §4 路线 D 的可编程边界缓解）；
- 机会成本：同等人力若投入 NPU 下一代的收益对比（建议在选型评审中显式回答）。

## 14. 推进路线 Roadmap

### Step 0（立项前，最高优先级）：Workload 实测

把 **DCCRN、GTCRN、Conv1D-KWS** 三个模型在现有 NPU 上真跑（或 cycle-accurate 仿真 / FPGA），产出报告：

- 各层利用率、逐帧延迟、整机功耗；
- STFT 以 matmul 实现的实际代价（n_fft=400 与 4096 两档）；
- DCCRN 权重带宽实测，校验 §7.1 的账。

**这份报告是立项评审的弹药，替换本报告中所有 ★ 标注的估计值。**

### Step 0.5：方案选型评审

以实测数据对比 §4 的 A/B/C/D 四条路线（PPA + 工程量 + 灵活性 + 机会成本），正式冻结路线。

### 算法团队先行交付（与 Step 0 并行，现在就可启动）

1. 冻结 §2.3 参数表（确认 320/400、COLA 约束、窗类型）；
2. 锁定 bit-exact 参考实现并生成 golden vectors；
3. 冻结 §11.2 各场景验收阈值；
4. Q15 STFT + INT16 RNN 定点链路仿真，验证 §8 精度假设；
5. 确定降噪模型主路线（GTCRN 优先 vs DCCRN 必选），驱动 §7.2 SRAM 档位决策。

### 分阶段交付

| 阶段 | 目标模型 | 交付 | 验收 |
|---|---|---|---|
| **Phase 1：AFE Engine** | 全部模型的 STFT/ISTFT/MFCC/重采样 | AFE RTL + 仿真器 | golden vectors bit-exact 全过；吞吐 ≥3.35M ops/s |
| **Phase 2：唤醒全链路** | Conv1D KWS | NN 执行单元 v1 + always-on 电源域方案 | 唤醒率达 §11.2 阈值；链路功耗 <1mW |
| **Phase 3：降噪** | GTCRN（主）/ DCCRN（视 §7 结论） | RNN 微码 + Conv1D 联动 | PESQ 达标；端到端 <10ms |
| **Phase 4：声纹 + 说话人日志** | ECAPA-TDNN / LSTM SD | 全功能 | EER < 阈值；DER < 阈值 |
| **Phase 5：NPU 联合验证** | SepFormer / ASR / TTS | AFE↔NPU 全链路 + 图切分工具 | 分离 SDR、ASR WER 达标；并发场景（降噪+ASR 同跑）演示 |

> Phase 1 之所以是第一刀：需求覆盖 100% 场景、数学定义永不过时、bit-exact 验收无争议空间、且不依赖 Conv1D/RNN 引擎边界的谈判结果。

## 15. 与 NPU 团队的协作要点

### 15.1 核心信息（对齐认知）

1. "我们要的是 **Audio Core，不是 Audio DSP**"——不做传统信号链，只做 AI 推理路径；
2. **AFE Engine 无争议**——所有模型必经、NPU 做不划算，先做它，不卡在引擎边界谈判上；
3. **Conv1D/RNN 引擎边界用 Step 0 实测数据裁决**，不用直觉争论；
4. **Transformer 不碰**——NPU 强项；Audio Core 与 NPU 是互补而非竞争。

### 15.2 立 RTL 前必须冻结的接口（前版 4 条 + 新增 2 条）

1. 数据格式协议：每个引擎输入/输出的 Q15/INT8/FP16 定义；
2. 握手协议：ready-valid vs 帧中断；
3. Buffer 规格：帧数、乒乓 vs 多 buffer；
4. 共享 SRAM：容量、bank 数、通道、AXI 接口；
5. **（新增）requant 归属**：Audio Core Q15 输出 → NPU INT8 输入的量化转换由哪一侧硬件完成；
6. **（新增）图切分工具 owner**：NPU 编译器扩展还是新建工具链，人力从哪出。

## 17. 结论

| 问题 | 结论 |
|---|---|
| **要不要做 Audio Core？** | **要。** 三条硬理由：always-on 功耗域鸿沟（最不可辩驳）、流式执行模型系统性错配、前端音质的 INT16 精度刚性 |
| **怎么做？** | **路线 D**：AFE 硬加速器（FFT/FIR/mel/LUT，数学永不过时）+ 可编程 NN 执行单元（兜底模型换代），拒绝全硬化 |
| **第一刀？** | **AFE Engine**：100% 场景覆盖、bit-exact 验收无争议、不依赖引擎边界谈判 |
| **最大技术账？** | **DCCRN 权重带宽（~1.2GB/s 若走 DRAM）**，而非 hidden state；决定 SRAM 档位与模型主路线 |
| **最大非技术风险？** | 软件工具链人力低估 + 未实测就立项 |
| **下一步？** | **Step 0 实测**（DCCRN/GTCRN/KWS 在现有 NPU 上跑，出利用率/延迟/功耗报告）+ 算法团队五项先行交付并行启动 |
