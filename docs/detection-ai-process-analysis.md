# 检测算法开发流程 AI 化分析与落地建议

## 1. 背景与问题定义

你当前负责的是芯片公司的检测类算法开发工作。团队需要与软件团队配合，为客户提供带检测功能的芯片能力。在实际交付中，客户会持续反馈误检和漏检问题，团队再基于客户数据进行优化。

当前痛点并不只是模型效果本身，而是从**拿到客户问题数据**到**定位问题、筛选和标注数据、训练新模型、量化转换、板端验证、整理交付**这一整条链路很长，人工参与点很多，导致问题解决周期较长。

因此，需要客观回答以下几个问题：

1. 整套流程能否完全 AI 化？
2. 哪些环节可以自动化，哪些环节必须保留人工决策？
3. 真正的困难点在哪里？
4. 如果建设一套内部系统，需要多少人力投入？
5. 如果现在开始做，第一步最应该做什么，才能最快解决痛点？

---

## 2. 当前流程概述

结合你的描述，当前检测算法迭代流程可整理为以下链路：

1. **工具与环境**
   - 训练服务器：用于模型训练和数据处理，限制性上网
   - Linux 开发板：带 NPU，用于测试和仿真
   - 转换工具 Docker 环境：用于模型量化、转换和导出

2. **问题定位**
   - 先判断是量化精度问题，还是浮点模型本身问题
   - 若是浮点模型问题，再进一步分析误检和漏检的具体类型

3. **数据筛选与标注策略制定**
   - 根据误检/漏检类型制定筛选规则和标注策略
   - 例如是否标注椅子/盆栽、抖动场景标多少帧、哪些重叠或远距样本不标等

4. **手工标注与数据合并**
   - 对筛选后的数据人工标注
   - 与原有训练数据集合并，形成统一的 `images + labels`

5. **训练**
   - 登录受限服务器
   - 在既定工程与 conda 环境中启动训练

6. **评估**
   - 在新增测试集和历史专业测试集上验证效果
   - 不达标则回到问题定位阶段继续迭代

7. **量化、转换与仿真**
   - 导出 ONNX 等中间格式
   - 进行量化、转换、仿真和量化精度评估

8. **板端验证**
   - 若代码不改，只在板端测试结果
   - 验证误检或漏检是否解决

9. **代码编译与板端联调**
   - 若代码有改动，则需要重新编译
   - 再在板端跑数据、看画框、验证效果

10. **差异对比与进一步定位**
    - 若效果仍不达标，需要比较仿真工具与板端结果
    - 必要时逐层比对定位精度损失来源

11. **整理与提交**
    - 整理代码、编写 readme 和文档
    - 提交到 Gerrit

## 3. 对两份附件分析的客观评价

附件中有两份材料：

- 一份是“原始 AI 化分析报告”
- 一份是“对原始报告的评审稿”

综合判断：

> **评审稿整体上比原始分析稿更贴近真实工程情况。**

但两份报告都各有长处，也各有偏差。

### 3.1 原始分析稿的合理之处

原始稿最合理的地方在于大方向判断：

- **全流程无法完全 AI 化**：这个结论是正确的
- **人负责决策、机器负责执行**：这是非常适合你们场景的原则
- **不建议一开始做大而全平台**：也符合实际投入产出比

这些结论适合用于和管理层沟通整体方向。

### 3.2 原始分析稿的问题

原始稿把部分环节说得过于悲观，尤其是下面三点。

#### （1）把“问题定位”整体自动化率判断得过低

问题定位其实应该拆成两类：

- **量化精度定位**
  - 浮点与定点结果比对
  - 逐层输出差异分析
  - 余弦相似度、KL 散度等指标计算
  - 问题层自动标记

  这部分本质上是规则明确、指标明确的工程问题，**自动化潜力很高**。

- **浮点模型本身的语义归因**
  - 比如椅子被误检成人
  - 目标靠近其他物体导致误检
  - 远距离、遮挡、姿态变化导致漏检

  这部分才是真正难自动化的地方，因为它涉及训练数据、标注方式、场景分布、策略取舍等复杂判断。

所以，不能把整个“问题定位”简单定义为低自动化。

#### （2）把“标注策略”整体定义为不可自动化

标注相关工作也必须拆开看：

- **策略定义**：难自动化
  - 哪些场景该标、哪些不该标
  - 某类误检值不值得修
  - 某类样本纳入训练会不会引起过拟合

- **策略执行与质检**：适合自动化
  - IoU 规则检查
  - 标注一致性检查
  - 抖动帧抽样
  - 视频抽帧去重
  - 冲突标注高亮

因此更准确的说法是：

> 标注策略本身依赖经验，但策略执行和标注质检可以大量自动化。

#### （3）把板端问题说成“非确定性 Bug”，表述不够严谨

更贴切的说法是：

- 困难不一定来自“随机非确定性”
- 更常见的是**仿真器与真实板端实现存在保真度差异**
- 也可能是量化、转换、算子实现、数据布局、驱动或代码逻辑差异

这意味着：

- **差异检测过程**可以自动化
- **差异根因判断**仍然需要经验工程师参与

### 3.3 评审稿的合理之处

评审稿有三个关键纠偏，基本都成立：

1. **把 RCA 拆成“量化诊断”和“语义归因”**
2. **把标注相关工作拆成“策略定义”和“策略执行/质检”**
3. **把板端差异纠正为“保真度或实现差异问题”**

这些拆分更符合真实工程流程，也更适合拿来做系统设计。

### 3.4 评审稿仍需谨慎的一点

评审稿提到多模态大模型可以辅助 badcase 归因，这个方向本身没有问题，但在你们场景里不能高估：

- 你们面对的是垂直客户场景，不是通用互联网图像
- 需要的是可追责的工程标签，不是泛泛解释
- 客户数据通常涉及保密和限制性网络，外部模型不一定能直接使用

因此，多模态模型更适合作为**后续增强能力**，不应作为第一阶段系统建设的核心。

## 4. 这套流程能否完全 AI 化？

## 结论

> **不能完全 AI 化到无人值守，但可以高度自动化，并叠加 AI 辅助决策。**

这里要明确区分两个概念：

### 4.1 “完全 AI 化”

指的是系统能够自动完成以下全链路动作：

- 接收客户数据
- 自动判断根因
- 自动决定标注策略
- 自动组织训练
- 自动量化和转换
- 自动板端验证
- 自动判断是否达到交付要求
- 自动输出最终模型给客户

对于你们这种芯片检测算法场景，这种“端到端无人化”并不现实。

### 4.2 “高度自动化 + AI 辅助”

更现实的目标是：

- 自动完成重复性执行工作
- 自动生成差异和回归报告
- 自动给出初步诊断建议
- 把人的精力集中到少数关键决策点上

这条路线是现实可行、且真正有价值的。

## 5. 为什么不能完全 AI 化：真正的困难点

### 5.1 困难点一：很多关键判断本质上是“取舍决策”

比如：

- 人和椅子有 IoU 时不标，没 IoU 时分别标
- 远距离复杂背景样本不要
- 高重叠目标不标
- 某类抖动画面只标一部分
- 某类长尾误检是否值得专门修

这些并不是纯客观的数学问题，而是带有业务取舍的策略判断：

- 修这类误检，会不会引入另一类漏检？
- 某类样本是否会把模型带偏？
- 这个问题是否值得为客户专项优化？

AI 可以辅助分析，但很难独立承担这种 trade-off 决策责任。

### 5.2 困难点二：客户 badcase 通常是长尾、小样本、信息不完整

客户反馈回来的数据常常具有以下特点：

- 数据量少
- 上下文不完整
- 可能只有截图或短片段
- 缺少完整复现条件
- 版本、阈值、运行环境信息不一定齐全

在这种情况下，系统很难直接给出确定根因，只能给出概率性建议。

### 5.3 困难点三：问题不只是模型问题，而是系统级问题

在你们流程里，一个误检/漏检可能来自：

- 浮点模型本身
- 训练数据选择
- 标注策略
- 量化校准集
- 某层量化敏感
- ONNX 导出或模型转换
- NPU 算子实现
- 板端前后处理代码
- 编译参数或运行时配置

所以这不是单纯的“训练模型自动化”问题，而是**模型、工具链、硬件、软件协同问题**。

### 5.4 困难点四：很多经验知识是隐性的

当前流程中，大量关键经验存在于工程师脑子里，例如：

- 哪类 FP 值得修
- 哪类 FN 不建议标
- 哪类视频要抽帧
- 哪类样本容易过拟合
- 哪类历史数据可能引入污染

如果这些知识没有被结构化沉淀，就很难做成稳定系统，更无法直接交给 AI 使用。

### 5.5 困难点五：最终交付目标不是“指标提升”，而是“客户问题真正解决”

自动化系统很容易优化：

- mAP
- precision / recall
- FP / FN 统计
- 量化误差

但客户真正关心的往往是：

- 某类抱怨的误检是否消失
- 某类关键漏检是否被修复
- 特定安装角度、特定背景下是否稳定

也就是说，最终的“达标”具有业务语义，而不仅仅是指标语义。

## 6. 哪些环节适合自动化，哪些环节必须保留人工

| 流程环节 | 内容 | 自动化潜力 | 建议形态 |
|---|---|---:|---|
| 环境准备 | 训练环境、转换环境、资源申请 | 高 | 工程自动化 / CI |
| 量化诊断 | 浮点 vs 定点、逐层 diff | 很高 | 规则引擎 + 自动报告 |
| 语义归因 | 误检/漏检原因初判 | 中 | AI 辅助 + 人工确认 |
| 标注策略定义 | 标什么、不标什么 | 低 | 人工主导 |
| 标注规则执行 | IoU 检查、抽帧、质检 | 高 | 脚本化 / 规则引擎 |
| 数据预处理与格式转换 | 合并、转换、版本化 | 高 | 自动流水线 |
| 训练调度 | 启动训练、保存模型、记录参数 | 高 | MLOps |
| 测试评估 | 专业测试集回归 | 高 | 自动回归报告 |
| 量化与转换 | 自动量化、导出、仿真 | 中高 | 流水线 + 风险告警 |
| 板端批量测试 | 批量推理、抓图、收集日志 | 高 | 自动板测 |
| 板端差异对比 | 仿真与板端逐层比较 | 高 | 自动 diff 报告 |
| 最终是否交付 | 是否满足客户问题诉求 | 低 | 人工拍板 |

总结起来：

> **可大幅自动化的是执行链；难以完全 AI 化的是判断链。**

## 7. 系统建设的人力投入预估

### 7.1 轻量闭环版

目标：不做大平台，先把最痛的流程打通。

典型能力包括：

- badcase 入库规范
- 自动量化诊断
- 自动回归测试
- 自动板端跑测
- 报告自动生成
- 历史 case 检索

**建议投入：6~10 人月**

适合先验证收益。

### 7.2 完整平台版

目标：形成“数据回流 - 诊断 - 训练 - 量化 - 板测 - 审批”的内部平台。

典型能力包括：

- 数据入库与版本管理
- 规则引擎
- 标注质检
- 训练调度
- 量化看板
- 板端实验室调度
- 新旧模型效果对比
- Web 审批看板
- 知识库与案例库

**建议投入：18~25 人月**

如果 NPU 工具链较封闭、基础设施不规范、历史数据分散，则实际投入可能进一步增加。

### 7.3 为什么很多方案容易低估成本

最容易漏算的成本通常有三类：

1. **规则知识沉淀成本**
   - 把工程师经验变成可配置规则非常费时

2. **NPU 工具链不透明带来的试错成本**
   - 文档不足、工具封闭、调试链路长

3. **长期运维成本**
   - 场景扩展、规则维护、流程升级都需要持续投入

## 8. 如果我是决策者，第一步具体该怎么做？

## 核心判断

如果现在只能优先做一件事，我不会先做“大而全平台”，也不会先做“自动标注”或“智能 Agent”。

> **第一步最应该做的是：客户 badcase 标准化入库 + 自动诊断/回归闭环。**

这一步最能直接打到当前痛点：**从拿到客户数据到明确下一步动作太慢**。

### 8.1 为什么第一步必须做这件事

当前真正拖长周期的，不只是训练慢，而是前面这些环节耗时很大：

- 客户数据格式不统一
- 人工整理数据
- 人工分别跑浮点、量化、板端结果
- 人工做差异对比
- 人工判断问题该走哪条修复路径
- 人工整理报告给后续训练或修改使用

如果这一段不标准化，后面即使把训练和板测都自动化了，整体周期也不会真正缩短很多。

### 8.2 第一阶段系统的最小闭环能力

建议第一步做成一个最小闭环产品，至少包含以下五项能力。

#### 能力 1：客户 badcase 标准化入库

统一问题包格式，至少记录：

- 客户 / 项目 / 芯片版本
- 模型版本
- 板端软件版本
- 原始图像或视频
- 当前检测结果
- 期望结果
- FP / FN 类型
- 是否可复现
- 相关日志或输出文件

目的：先解决“问题描述不标准、问题资产散落”的问题。

#### 能力 2：自动分流到三条诊断路径

针对同一份 badcase，系统自动跑：

1. 服务器浮点推理
2. 量化/仿真推理
3. 板端推理

然后自动输出分流结论，例如：

- float 正常、quant 异常
- float 异常、quant 同样异常
- float 和 quant 正常，但 board 异常
- 三者一致，但仍不符合业务预期

这样可以快速判断问题优先归属到：

- 数据 / 训练问题
- 量化问题
- 板端实现问题
- 阈值或业务规则问题

#### 能力 3：自动生成差异报告

报告不只给指标，还应给工程师可直接使用的信息：

- 新旧模型 FP / FN 对比
- float / quant / board 结果并排对比
- 关键层误差摘要
- 相似历史案例推荐
- 可能根因标签建议

注意：这里的建议是“辅助判断”，不是替代最终决策。

#### 能力 4：自动回归测试

每次训练新模型、量化新模型或修改代码后，自动在以下数据上回归：

- 本次客户 badcase 集
- 历史已知问题集
- 专业测试集

并明确给出：

- 修复了哪些 case
- 回归坏掉了哪些 case
- 哪些 case 只在板端掉点

这一步可以显著减少“修一个坏两个”的风险。

#### 能力 5：历史案例知识库

把每次问题最终的结论和处理动作沉淀下来：

- 问题现象
- 根因归类
- 最终解决方式
- 对应数据策略
- 是否涉及量化
- 是否涉及代码改动
- 是否涉及板端差异

这是后续做 AI 辅助系统最重要的地基。

### 8.3 这第一步解决的核心痛点

如果上述最小闭环完成，最直接解决的是：

1. **缩短“拿到客户数据到明确处理方向”的时间**
2. **减少工程师在重复执行上的时间消耗**
3. **把经验从个人脑中转化为可复用资产**
4. **建立后续 AI 辅助决策所需的数据基础**

## 9. 推荐的落地路线图

### 阶段 1：先固化问题闭环

优先实现：

1. badcase 入库规范
2. float / quant / board 自动三路推理
3. 自动差异报告
4. 自动回归测试
5. 历史 case 检索

这是性价比最高的阶段，建议优先投入。

### 阶段 2：沉淀规则知识

在第一阶段稳定后，再建设：

1. 标注规则引擎
2. 标注质检
3. 抽帧 / 去重 / 聚类
4. 数据筛选辅助
5. 场景标签体系

这一阶段会显著减少数据侧的人力消耗。

### 阶段 3：再引入 AI 助手

最后再建设：

1. badcase 自动初判
2. 相似案例召回
3. 训练和量化参数推荐
4. 交互式决策助手

这样 AI 才是建立在真实数据与规则积累上的，而不是一开始就空转。

## 10. 最终结论

### 10.1 对两份附件的综合判断

- 原始分析稿：方向对，但把部分可自动化环节说得过于悲观
- 评审稿：整体更专业、更符合真实工程情况
- 但对于多模态大模型的短期效果仍应保持谨慎预期

### 10.2 对“能否完全 AI 化”的最终判断

> **不能完全 AI 化到无人值守。**

但可以实现：

> **执行链高度自动化 + 决策链 AI 辅助化。**

### 10.3 如果现在就开始做，最值得投入的第一步

> **优先建设“客户 badcase 标准化回流 + 自动诊断/回归闭环”。**
