# 算法自动化工具流程修订版方案

**文档类型：** 建设方案（修订版）  
**版本：** v1.0  
**日期：** 2026-07-09  
**修订基础：** 《算法自动化工具流程详尽方案.docx》  
**适用对象：** 算法 / 软件 / IT / 项目负责人

---

## 0. 修订原则

1. **保留原方案正确骨架**，只改不合理处，不另起炉灶。  
2. **先证据、后分流、再决策**；大模型只做初筛辅助，不做最终归因。  
3. **执行自动化，决策人工化**；每个人工节点必须有进入、证据包、结构化表单、退出回路。  
4. **平台无关主干 + V821/V861/V853 适配器**。  
5. **分两期建设**：一期打通分诊闭环，二期打通训练到交付闭环。

---

## 1. 建设目标与边界

### 1.1 目标

建设内部算法自动化工具，缩短两类痛点：

| 痛点 | 目标 |
|---|---|
| A. 从拿到客户数据到明确下一步动作太慢 | 标准化入库、自动两路推理、规则分诊、报告与派单 |
| B. 模型训练好后同步给客户的闭环太长 | 训练评估、三平台转换、板端验证、代码/内存/Gerrit、内部候选交付 |

### 1.2 非目标（一期不做）

- 端到端无人值守自动交付客户
- 用大模型替代人工做最终归因/标注策略决策
- 自动超参搜索
- 完整 Web 大平台（一期可用脚本 + 任务单 + 报告）
- 一次兼容所有历史项目的全部特殊流程

### 1.3 建设分期

| 期次 | 范围 | 建议人月 |
|:---:|---|:---:|
| 一期 | 分诊闭环（入库→两路推理→差异分析→归因初筛→报告派单） | 5~7 |
| 二期 | 标注质检、合并训练、三平台转换、板测、代码/内存/Gerrit、候选交付 | 8~12 |
| 合计 | 可复用骨架上的完整闭环 | 13~19 |

三平台适配额外预留 **+1.5~3 人月**（V853 外部工具链不确定性最大）。

## 2. 总体架构

### 2.1 两层闭环

```text
闭环 A：问题分诊
1号通拉数 → 校验入库 → 版本对齐 → 浮点/板端推理
→ 差异分析 → FP/FN判定 → 归因初筛 → 人工确认 → 任务单

闭环 B：修复与交付
数据筛选标注 → 合并训练 → 浮点评估
→ ONNX基线 → 平台转换(V821/V861/V853)
→ 仿真对比 → 板端验证 → 代码/内存/Gerrit
→ 内部候选release → 审批 → 客户同步 → 反馈回写
```

### 2.2 最小公共骨架

| 组件 | 作用 |
|---|---|
| Case / 任务单 | 全流程主键 |
| 制品管理 | 数据、模型、转换产物、报告、文档 |
| 元数据 | 客户、平台、版本、问题类型、状态、责任人 |
| 执行适配器 | 训练服务器 / 转换 Docker / 板端 |
| 规则库 | 标注规则、分诊规则、门禁阈值 |
| 状态机 | 自动运行 / 等待人工 / 恢复自动 / 外部等待 |
| 报告引擎 | 分诊报告、训练报告、平台报告、板测报告 |

### 2.3 统一任务状态机

```text
created
 → validating
 → auto_running
 → waiting_manual
 → manual_in_progress
 → decision_submitted
 → auto_resumed
 → external_waiting   # 主要用于 V853 芯原支持
 → blocked
 → resolved_internal
 → released_candidate
 → customer_synced
 → closed
 → known_limitation   # 场景无解归档
```

人工干预必须走：

```text
自动暂停 → 派单 → 证据包 → 结构化决策 → 自动从断点继续
```

## 3. 闭环 A：从 1 号通拉数到明确下一步

### 3.1 数据完整性校验（自动）

```text
□ 文件可解压/可读（图片格式、视频编码）
□ 无损坏文件、无 0 字节文件
□ 必填元数据齐全：
   - 客户/项目名
   - 平台型号（V821 / V853 / V861）
   - 驱动/模型/代码版本
   - 输入分辨率/数据格式
   - 问题描述（误检/漏检、期望结果）
□ 数据量统计（图/视频/时长）
```

不合格 → 自动生成补充清单，退回项目对接人。

### 3.2 标准化入库（自动）

- 图片/视频唯一 ID
- 统一目录结构
- 元数据结构化记录
- 视频抽帧、图片去重
- 抽帧结果与原视频建立映射

输出：标准 case 包 + 可推理数据清单。

### 3.3 版本对齐检查（自动）

- 客户模型版本 vs 内部主线版本
- 同客户/平台/问题类型历史 case 检索

若客户版本落后且问题已在新版本修复：

> 直接生成“建议升级版本”回复，流程结束，不进入分析。

### 3.4 前后处理对齐后，自动跑两路基准

比对前必须对齐：

- resize
- 归一化
- 颜色通道
- NMS
- 阈值

然后自动执行：

1. 服务器浮点推理 → 标签、置信度、框、可视化  
2. 对应平台板端推理 → 框、日志、可视化  

### 3.5 规则化差异分析（自动）

| 现象 | 初步分流 | 派单方向 |
|---|---|---|
| 浮点有框、板端没框 / 置信度差异大 | 疑似量化/转换/前后处理问题 | 先自动逐层诊断，再派算法或软件 |
| 浮点和板端都错 | 疑似模型/数据问题 | 算法 |
| 浮点和板端都对，客户仍报问题 | 疑似版本/配置/阈值/复现问题 | 项目对接人 |
| 与历史 case 高相似 | 建议复用历史路径 | 对应责任人 |

附加自动标签：

- 置信度分布
- 卡阈值框占比
- 目标尺寸分布
- 框重叠度
- 逐帧抖动幅度

### 3.6 分诊报告 + 人工确认

报告至少包含：

- case 基本信息、版本对齐结论
- 浮点/板端并排可视化
- 差异统计与分流结论
- 历史相似 case
- 建议路径（量化排查 / 数据训练 / 版本核对）

人工确认后生成任务单，进入对应路径。

## 4. 浮点问题路径：判定、归因、筛选、标注

> 仅当确认“浮点模型本身复现客户问题”后进入本段。

### 4.1 FP/FN 互斥判定（修订后）

```text
对每个检测框，取与真值最大 IoU：
1) IoU ≥ 0.5 且类别对 → 正确
2) IoU ≥ 0.5 且类别错 → 分类错误
3) 0.1 ≤ IoU < 0.5 → 定位偏移
4) IoU < 0.1 → 误检 FP

对每个真值框：
5) 无 IoU ≥ 0.5 的检测框：
   - 存在 0 < conf < 板端阈值 的匹配框 → 卡阈值型漏检
   - 完全无框 → 漏检 FN
6) 同一真值匹配多个检测框 → 重复框
```

#### 自动化程度

- **有基准时半自动**：脚本完成统计，但基准可疑，指标仅供参考  
- **无基准时半自动**：可视化点选“此框误检 / 此处漏了”，形成最低成本基准  

#### 假 FP 检查（必做）

人工二判时必须确认：

> 该位置是否真的没有目标？

若有目标 → 改判“基准漏标”，修基准，不进模型修复。

#### 非数据类结果去向

| 判定 | 去向 |
|---|---|
| 重复框 | NMS/后处理，不进训练 |
| 分类错误 | 类别标注/混淆负样本路径 |
| 定位偏移 | 标注质量或回归困难路径 |
| 卡阈值型漏检 | 先试阈值/后处理，再决定是否补数据 |

### 4.2 误检原因表（保留并增强）

| 编号 | 原因 | 典型表现 | 诊断信号 |
|---|---|---|---|
| b.1 | 背景标注污染 | 椅子/盆栽被稳定检成人 | 训练集溯源；误检位置固定、置信度稳定 |
| b.2 | 近邻关联误检 | 人靠近时物体出框，离开后消失 | 误检框与人框距离统计 |
| b.3 | 场景缺失误检 | 条状物/全图框/抖动误检 | 长宽比、框面积占比、逐帧方差 |
| b.7 | 类人真实物体 | 模特/海报/屏幕中的人 | 人工确认；修负样本或业务规则 |
| b.8 | 成像条件误检 | 暗光/过曝/IR 噪声出框 | 亮度/信噪比分组统计 |
| b.9 | 训练分布外场景 | 新镜头/俯视/鱼眼误检偏多 | 按客户/镜头分组统计 |
| b.10 | 负样本不足 | 背景物体零星误检 | 训练集背景图占比 |
| b.11 | 标注不一致 | 置信度摇摆、卡阈值 | 同类场景标注一致性检查 |

### 4.3 漏检原因表（补 b.16）

| 编号 | 原因 | 典型表现 | 诊断信号 |
|---|---|---|---|
| b.4 | 远距离小目标 | 小尺寸召回骤降 | FN 尺寸分布集中在小目标 |
| b.5 | 重叠/遮挡 | 重叠时只出一个框 | 邻近 IoU；NMS 前后框数对比 |
| b.6 | 姿态变化/卡阈值 | 蹲弯侧漏检；0.4~0.6 摇摆 | 置信度分布定位阈值或特征问题 |
| b.12 | 罕见姿态 | 躺/倒地/爬行漏检 | 训练集姿态分布空白 |
| b.13 | 边缘截断 | 半身/头肩漏检 | FN 位置集中在边缘 |
| b.14 | 成像退化 | 暗光/逆光/模糊/雨雾漏检 | 亮度/清晰度分组召回 |
| b.15 | 低对比 | 衣着与背景相近漏检 | 人工归纳为主 |
| **b.16** | **输入分辨率/前处理损失** | **原图浮点能检，板端分辨率 resize 后浮点也漏** | **用板端相同前处理重跑浮点；先于 b.4 排除** |

### 4.4 大模型初筛约束

大模型只做初筛，输出必须满足：

1. 标签只能是 `b.1~b.16` 或 `无法归类`
2. 必须附诊断信号数值
3. 支持多标签
4. 生成结构化 Log 给人二判
5. 人工改判结果回写，持续修正匹配规则

Log 建议格式：

```text
case_id | 判定(FP/FN/卡阈值/定位/重复框) | 建议归类(b.x，可多选)
| 证据指标 | 置信度(高/中/低) | 建议下一步
```

### 4.5 数据筛选与标注策略（修订后）

#### 误检类

| 原因 | 要什么 | 不要什么 | 标注方式 |
|---|---|---|---|
| b.1 | 先溯源清洗旧污染标注；补无人家具负样本；人椅无 IoU 时分别标注 | 人椅有 IoU 的场景 | 人工 |
| b.2 | 人与物体靠近但无交集，精确分别标注 | 边界人眼难分的 | 人工 |
| b.3 条状物 | 灯杆/衣架等负样本 | — | 自动收集，人工确认无目标 |
| b.3 抖动 | 静止场景约 100 张；抖动严重挑视频连续帧 | 同场景超量重复帧 | 首帧人工，后续帧传播，人工抽检 |
| b.7 | 模特/海报/屏幕负样本或单独类 | — | 人工（先业务决策是否修） |
| b.10 | 各类纯背景图 | — | 人工确认无目标后入库 |
| b.11 | 不补新数据，先清洗旧标注一致性 | — | 人工 |

#### 漏检类

| 原因 | 要什么 | 不要什么 | 标注方式 |
|---|---|---|---|
| b.4 | 干净背景远距样本 | 远距+复杂背景 | 高分辨率预标注+人工补漏；否则纯人工 |
| b.5 | 中低重叠、边界可辨样本 | 高重叠难分割样本 | 人工 |
| b.6 | 蹲/弯/侧等多场景姿态 | 单场景大量同姿态 | 人工；卡阈值先试后处理 |
| b.12 | 躺/倒地专项数据 | — | 人工专项 |
| b.13 | 出入画连续帧 | — | 人工关键帧 + 帧传播 |
| b.14 | 对应光照/天气真实样本 | 人眼不可辨样本 | 人工；不可辨转“场景无解” |
| b.15 | 同类低对比真实样本 | 人眼不可辨样本 | 人工 |
| b.16 | 不补数据 | — | 改输入设定/多尺度方案 |

### 4.6 标注三档分工

| 档位 | 范围 | 方式 |
|---|---|---|
| 一档可自动 | 高置信常规样本、帧传播、已确认纯背景入库 | 自动 + 抽检 |
| 二档半自动 | 中等难度、中低重叠、一般姿态 | 预标注 + 人工修正 |
| 三档必须人工 | b.1 相关、远距小目标、高重叠丢弃决策、负样本无目标确认、卡阈值、新场景首批、旧污染清洗 | 人工 |

原则：

> badcase 核心难例不能依赖自动标注；自动标注主要服务周边扩充数据和重复劳动。

## 5. 数据合并、训练、评估与不达标回环

### 5.1 合并前自动检查

```text
□ 格式检查：YOLO 标签、类别 ID、坐标 [0,1]
□ 完整性：image/label 对应；负样本空 label 需显式声明
□ 越界检查：越界框、零面积框、极端长宽比
□ 去重：与现有训练集哈希/相似度去重
□ 泄漏检查：同一视频帧不得同时进 train/val
□ 规则符合性：IoU 规则、抖动 100 张上限等
```

不过则打回标注。

### 5.2 badcase 数据三份去向（修订）

| 去向 | 建议 | 用途 |
|---|---|---|
| 训练集 | 70~80% | 学习 |
| badcase 验证集 | 20~30%（小样本可更高） | 验证本次是否修复，永不进训练 |
| 历史回归集 | 问题解决后归档代表样本 | 防止未来复发 |

必须按视频/场景划分，禁止按帧随机划分。

### 5.3 合并配比与版本登记

半自动确认：

- badcase 采样倍率（常见 2~5x）
- 负样本占比上限
- 合入前后分布变化报告

全自动登记 manifest：

```text
dataset_vYYYY.MM.DD_caseXXX
  基线 / 新增 / 修改 / 删除
  来源 case 与 b.x 分布
  采样倍率
  train/val 划分方式与种子
```

### 5.4 训练配方模板（修订为 4 个）

| 模板 | 场景 | 要点 |
|---|---|---|
| A 增量修复 | 少量新数据，未改旧标注 | finetune，低 LR，较少 epoch |
| B 清洗重训 | 做了 b.1/b.11 旧标注清洗 | 全量重训 |
| C 小目标专项 | b.4 为主 | 提高输入分辨率/多尺度 |
| D 负样本强化 | b.1/b.3/b.10 为主 | 提高负样本采样权重 |
| QAT 模板 | PTQ 无法达标 | QAT 训练后重新导出 ONNX |

人工节点：启动前确认模板；改超参必须写理由。

### 5.5 Loss 监控（自动）

| 信号 | 动作 |
|---|---|
| NaN/Inf | 自动停训，反查最近 batch |
| 持续不降 | 告警 |
| 周期性突刺 | 反查新合入脏数据 |
| train 降、val 不升反降 | 建议降采样/早停 |
| 高 loss 样本榜 | 回标注复核 |

输出：训练报告（分项 loss、val 曲线、异常点、高 loss 清单、与上版对比）。

### 5.6 三层测试与门禁（修订）

| 层 | 问题 |
|---|---|
| badcase 验证集 | 本次问题修好了吗 |
| 历史回归集 | 有没有修一个坏一个 |
| 专业测试集 | 大盘是否掉点 |

门禁示例（训练前写入任务单）：

```text
① badcase 修复率 ≥ 80%；关键客户 case 100%
② 历史回归通过率 100%
③ 专业测试集 mAP 下降 ≤ 0.5%，关键 Recall 下降 ≤ 1%
```

判定自动，结论人工确认。

### 5.7 不达标回环

| 类型 | 回到 | 动作 |
|---|---|---|
| A 本次没修好 | 数据/归因 | 补数据、重定位、继续清洗 |
| B 修好但大盘掉 | 配比/模板 | 降采样、改 finetune，不重标 |
| C 历史复发 | 规则一致性 | 清洗新旧规则冲突样本 |
| D 卡阈值 | 后处理 | 调阈值/NMS，不重训 |

纪律：

1. 每轮只改一类变量  
2. ≥3 轮升级负责人评审  
3. 每轮留档  
4. 允许“场景无解/部分无解”合法出口

## 6. ONNX 基线与三平台转换

### 6.1 ONNX 导出与基线校验

自动导出并保存：

- onnx 模型
- 输入尺寸、前后处理、类别表、阈值、NMS
- 训练数据版本、训练任务 ID

自动检查：

- 可加载、shape 正确、ORT 可推理
- ONNX vs 训练框架浮点结果一致（框/分/类）

不一致 → 人工修导出 → 重导出 → 重校验。

### 6.2 平台分发

ONNX 基线通过后，按目标平台扇出：

- V821
- V861
- V853

各平台独立配置、独立产物、独立报告。

### 6.3 V821

```text
ONNX
 → 截断/子图切分 → 截断后浮点校验
 → PTQ → 仿真 vs ONNX
 → 达标？
    是 → main_backend 参数确认 → 预编译 → 模型 + .c/.h → 板端验证
    否 → 混合量化 / 换校准集 / QAT回训练 / 调截断 / 工具链提单
```

关键人工点：

1. 截断点与 NPU/CPU 划分  
2. PTQ 不达标策略  
3. main_backend 参数确认  

### 6.4 V861（自研 NPU）

```text
ONNX → 转换 → 量化 → 仿真器 vs ONNX
 → 达标？
    是 → 加密导出 → 板端验证
    否 → 调量化 / 换算子 / 提自研工具链 / 必要时回模型结构
```

优先把逐层 dump 自动比对做在 V861（可控性最高）。

### 6.5 V853（芯原 NPU）

```text
ONNX → 芯原转换 → 量化 → 仿真器 vs ONNX
 → 达标？
    是 → 模型导出 → 板端验证
    否 → 调量化/校准集/规避算子/提交芯原支持
```

必须支持状态：`external_waiting`。

### 6.6 统一仿真比对逻辑

最终检测：

- 框数、IoU、置信度跨阈值、类别、新增 FP/FN

逐层：

- cosine / max abs / mean abs / top-N 异常层

自动结论：

| 现象 | 结论 |
|---|---|
| 最终一致且逐层可接受 | 转换/量化通过 |
| 最终不一致且误差集中量化层 | 量化问题 |
| 最终不一致但逐层误差不大 | 前后处理/阈值/NMS 问题 |
| shape/layout 异常 | 转换/算子问题 |

## 7. 板端验证、代码合规与交付

### 7.1 模型板端验证（转换后先做）

三类强制对比：

1. 现有板端 vs 原板端  
2. 现有板端 vs 基准  
3. 现有板端 vs 仿真器  

测试集：

- 本次 badcase
- 历史回归
- 专业测试集

门禁：

- 客户问题修复标准
- 历史回归标准
- 大盘指标标准
- 性能标准（帧率/内存/加载/长稳）

### 7.2 是否涉及代码修改

| 情况 | 是否改代码 |
|---|---|
| 只替换模型 | 否 |
| 只改配置（视管理方式） | 可能否 |
| 改前后处理/NMS/runtime/main_backend/.c.h | 是 |

#### 分支 A：只更新模型

收集模型/配置/报告 → 生成 readme/release note → 内部候选包 → 审批。

#### 分支 B：涉及代码修改

```text
代码整理
 → 编译
 → 板端回归
 → 内存检查
 → 文档生成
 → Gerrit Review
 → 合入后生成内部候选包
```

### 7.3 内存检查（必做）

| 检查项 | 目的 |
|---|---|
| 模型加载峰值 | 防超限 |
| 推理中内存增长 | 查泄漏 |
| 多次初始化/释放 | 查资源释放 |
| 长时间循环推理 | 查慢性泄漏 |
| 异常/空输入 | 查异常路径 |

失败 → 软件/嵌入式修复 → 重编译 → 重测。

### 7.4 Gerrit 与文档

自动准备：

- 代码 diff、编译结果
- 内存检查报告、板测报告
- 模型版本信息
- readme.txt / doc / release note

提交门禁：

- 编译通过
- 内存检查通过
- badcase 通过
- 历史回归通过
- 文档完整
- 负责人审批

readme.txt 至少包含：版本信息、解决问题、使用方式、已知限制、验证摘要。

### 7.5 内部候选 release 与客户同步

```text
内部候选 release 包
 → 版本与报告归档
 → 算法+软件负责人审批
 → 进入客户同步流程
 → 客户确认解决/未解决
 → 回写 case 库 → closed
```

支持：

- 只发布问题平台
- 或多平台同步发布

禁止：板端刚过就自动“一号通直接发客户”。

| 训练启动确认 | 合并完成 | 算法 | 训练 |

## 10. 一期 MVP 建议

### 10.1 一期只做这些

1. 1 号通拉数后的校验/入库/版本对齐  
2. 浮点 + 板端自动推理  
3. 规则差异分析与分诊报告  
4. FP/FN 半自动判定  
5. 大模型初筛 Log（可选）+ 人工确认  
6. 历史 case 检索  
7. 结构化派单  

### 10.2 一期明确不做

- 自动标注替代人工核心难例
- 全自动训练决策
- 三平台转换全自动
- 自动发客户

### 10.3 一期验收

| 指标 | 目标 |
|---|---|
| case 可标准化入库比例 | ≥ 70% |
| 两路结果自动产出比例 | ≥ 80% |
| 分诊报告可直接用于决策比例 | ≥ 70% |
| “拿到数据→明确下一步”耗时下降 | 30%~50% |
| 算法/软件持续使用意愿 | 愿意继续用 |

## 11. 与原方案的主要差异摘要

| 原方案问题 | 修订版处理 |
|---|---|
| 章节把分诊/训练/部署混在一起 | 拆成 A/B 两层闭环 |
| 量化问题只派算法 | 先诊断再按原因派算法/软件/工具链 |
| IoU 判定区间重叠 | 改为互斥判定顺序 |
| 缺 b.16 | 补入并在 b.4 前排除 |
| b.4/负样本“自动标注”不当 | 改为高分辨率预标注或人工确认 |
| 验证集 10% 偏少 | 小样本 20~30% |
| 专业测试集未进门禁 | 补进达标标准 |
| 板测与代码分支顺序乱 | 先模型板测，再代码分支 |
| 缺状态机/MVP/人月 | 本方案补齐 |
| 交付无客户反馈回写 | 增加 closed 回环 |

## 12. 落地建议

如果现在就要开工，建议顺序：

1. **先落地闭环 A（5~7 人月）**  
2. 同步沉淀规则库与 case 状态机  
3. 再做闭环 B 的训练评估与三平台适配  
4. 最后做代码/内存/Gerrit/候选交付自动化  

一句话：

> **不要先追求“全自动发客户”，先追求“客户数据进来后，系统能稳定给出可执行的下一步”。**
