# 一、从 1 号通上拉完数据 → 做数据校验、标准化入库、版本对齐检测

## 1. 数据完整性校验

校验清单（脚本自动执行）：

- 文件是否可解压/可读取（图片格式、视频编码）
- 是否有损坏文件、0 字节文件
- 必填元数据是否齐全：
  - 客户/项目名
  - 平台型号（V821 / V853 / V861）
  - 驱动/模型/代码版本
  - 输入数据分辨率/数据格式
  - 问题描述（误检还是漏检、期望结果）
- 数据量统计（多少张图、多少段视频、总时长）

不合格时：自动生成补充信息清单，退回项目对接人，不进入后续分析。

## 2. 标准化入库

校验通过后，自动完成：

- 按照图片、视频进行唯一 ID 管理
- 生成统一目录结构
- 元数据写入结构化记录（包括日期、数据格式、平台、版本等）
- 视频抽帧处理、图片去重
- 抽帧后的数据统一管理，并与上述图片、视频建立对应关系
- 输出可直接用于浮点/板端推理的数据清单

## 3. 版本对齐检查

- 客户用的模型版本 vs 内部当前主线版本
- 该客户/该场景是否有历史 case（按客户、平台、问题类型查库）

如果客户版本落后，且该类问题在新版本已修复：

> 直接生成“建议升级版本”的回复，流程到此结束，不进入分析。

## 4. 自动跑两路基准信息

比对前先对齐前后处理配置：

- resize
- 归一化
- 颜色通道
- NMS
- 阈值

然后自动执行：

1. 浮点推理 → 标签、置信度、检测框、可视化
2. 对应平台的板端推理（V821 / V853 / V861 选对应板子）→ 检测框、日志、可视化

## 5. 规则化差异分析

浮点结果 VS 板端结果：

| 现象 | 初步判断 | 派单方向 |
|---|---|---|
| 浮点有框、板端没框（或置信度差异大） | 可能量化/转换/前后处理问题 | 先自动逐层诊断，再派算法或软件 |
| 浮点和板端都错（误检/漏检） | 可能模型/数据问题 | 派给算法 |
| 浮点和板端都对，客户仍报问题 | 可能版本、配置、阈值、复现问题 | 派给项目对接人 |
| 与历史 case 特征匹配（同客户+同平台+同问题类型） | 建议复用历史路径 | 附上历史处理记录 |

附加分析标签（自动计算）：

- 置信度分布
- 卡阈值附近的框占比
- 目标尺寸分布（远距小目标）
- 框重叠度（IoU）
- 逐帧框抖动幅度

## 6. 生成报告 + 人工确认

报告内容：

- case 基本信息、版本对齐结论
- 浮点/板端结果并排可视化
- 差异统计和规则分流结论
- 历史相似 case 及当时的处理方式
- 建议路径（量化排查 / 数据训练 / 版本核对）

派单给对应责任人做分诊确认，人工干预确认后进入下一环节。

# 二、模型训练

## 1. 派给算法的任务

### 1.1 浮点有框、板端没框

进入人工分析前，先自动完成：

- 逐层 dump 比对（余弦相似度 / 误差统计）
- top-N 问题层定位
- 前后处理是否一致检查

人工分析后分流：

| 结论 | 动作 |
|---|---|
| 量化/混合量化可解 | 修改量化配置，重新量化/仿真，再回自动化板端测试 |
| 需要 QAT | 回到训练环节做 QAT，再重新导出 ONNX |
| 前后处理/代码问题 | 人工修改代码并编译 → 给到自动化工具进行板端测试 |
| 工具链问题 | 提单挂起（V861 内部 / V853 芯原） |
| 暂无更好解决方法 | 上报反馈，程序终止或转已知限制 |

### 1.2 判定是误检还是漏检

先人工检查第一章第 6 步产生的基准有无误检和漏检，给出反馈，用于后续自动化结果提醒。

对每张 badcase 图，把检测结果和基准（客户期望结果，或人工快速确认的真值）做 IoU 匹配。

#### 互斥判定顺序（可脚本执行）

```text
对每个检测框，取与真值最大 IoU：
1) IoU ≥ 0.5 且类别对 → 正确
2) IoU ≥ 0.5 且类别错 → 分类错误
3) 0.1 ≤ IoU < 0.5 → 定位偏移
4) IoU < 0.1 → 误检 FP

对每个真值框：
5) 无 IoU ≥ 0.5 的检测框：
   - 存在 0 < conf < 板端阈值 的匹配框 → 卡阈值型漏检
   - 完全无框 → 漏检 FN
6) 同一真值匹配多个检测框 → 重复框（NMS/后处理问题，不是数据问题）
```

#### 判定自动化程度

- 有基准时不完全自动：客户给了期望结果，或人工先花几分钟标记“这张图哪里错了”，之后 FP/FN 判定、统计、分布分析全部脚本完成（基准本身可能有问题，指标仅供参考）
- 无基准时半自动：系统把检测结果可视化并排展示，人工一键点选“此框是误检 / 此处漏了”——这个动作本身就是最低成本的基准标注
- 假 FP 检查：人工二判时必须确认“该位置是否真的没有目标”；若有目标，改判为基准漏标，修基准而不是修模型

#### 非数据类结果去向

| 判定 | 去向 |
|---|---|
| 重复框 | NMS/后处理路径，不进数据训练 |
| 分类错误 | 类别标注 / 混淆负样本路径 |
| 定位偏移 | 标注质量或回归困难路径 |
| 卡阈值型漏检 | 先试阈值/后处理，再决定是否补数据 |

### 1.3 误检情况

实际场景会出现但不止如下情况。大模型反馈信息与如下进行匹配，梳理成 Log 信息反馈给用户，二次人工判断。

约束：

- 输出标签限定为下表编号或“无法归类”
- 必须附诊断信号数值
- 支持多标签
- 人工改判结果回写

| 编号 | 原因 | 典型表现 | 诊断信号（可自动计算） |
|---|---|---|---|
| b.1 | 背景标注污染：标人时把椅子/盆栽框进去 | 无人场景中椅子/盆栽被稳定检出为人 | 训练集溯源；误检位置固定、置信度稳定 |
| b.2 | 近邻关联误检：人和某物体靠近但无交集 | 人走近椅子时椅子出框，人离开后消失 | 误检框与人框距离统计 |
| b.3 | 场景缺失误检：类人条状物、全图框、抖动 | 灯杆/衣架/窗帘被检成人；全图大框；抖动闪烁 | 长宽比异常；框面积占比>90%；逐帧方差大 |
| b.7 | 类人真实物体：模特、海报、屏幕、镜面 | 商场模特、广告被检成人 | 需人工确认；修负样本或业务规则 |
| b.8 | 成像条件误检：暗光噪点、过曝、红外纹理 | 夜晚 IR 模式噪声区域出框 | 按亮度/信噪比分组统计 |
| b.9 | 训练分布外场景：鱼眼、俯视、新镜头 | 新客户设备上误检明显偏多 | 按客户/镜头分组统计 |
| b.10 | 负样本不足 | 各种背景物体零星误检 | 训练集背景图占比统计 |
| b.11 | 标注不一致 | 置信度摇摆、卡阈值附近 | 同类场景标注一致性检查 |

### 1.4 漏检原因

实际场景会出现但不止如下情况。大模型反馈信息与如下进行匹配，梳理成 Log 信息反馈给用户，二次人工判断。

| 编号 | 原因 | 典型表现 | 诊断信号 |
|---|---|---|---|
| b.4 | 远距离小目标 | 目标像素小于某尺寸时召回骤降 | FN 目标尺寸分布集中在小尺寸段 |
| b.5 | 目标重叠/遮挡 | 两人重叠时只出一个框 | 邻近 IoU；NMS 前后框数对比 |
| b.6 | 姿态变化 / 卡阈值 | 蹲、弯腰、侧身漏检；0.4~0.6 摇摆 | 置信度分布区分阈值问题或特征问题 |
| b.12 | 罕见姿态：躺、倒地、爬行 | 跌倒场景检不出 | 训练集姿态分布统计 |
| b.13 | 画面边缘截断目标 | 目标刚进画面时漏检 | FN 框位置集中在图像边缘 |
| b.14 | 成像退化：暗光、逆光、模糊、雨雾 | 特定时段/天气漏检率高 | 按亮度/清晰度分组统计召回 |
| b.15 | 目标与背景低对比 | 特定衣着漏检 | 需人工归纳，较难自动 |
| b.16 | 输入分辨率/前处理损失 | 原图浮点能检，按板端分辨率 resize 后浮点也漏 | 用板端相同前处理重跑浮点；在归为 b.4 前先排除 |

## 2. 误检类筛选及标注策略

| 原因 | 要什么 | 不要什么 | 数量控制 | 说明（人工/自动） |
|---|---|---|---|---|
| b.1 背景污染 | ① 先溯源清洗 11-12 训练集污染标注 ② 补无人但有椅子/盆栽负样本 ③ 人椅无 IoU 时分别标注 | 人椅有 IoU 的场景（不标，不入训练） | 负样本适量，避免淹没正样本 | 人工 |
| b.2 近邻误检 | 人与物体靠近但无交集，精确分别标注 | 边界模糊、人眼难分是否有交集的 | 同场景控制数量 | 人工 |
| b.3 条状物/全图框 | 含类人条状物的负样本 | — | 每种类型若干即可 | 自动收集，人工确认无目标 |
| b.3 抖动 | 静止画面约 100 张；抖动严重专门挑视频连续帧 | 同静止场景超量重复帧 | 100 张/静止场景 | 首帧人工，后续帧传播，人工抽检 |
| b.7 类人真实物体 | 模特/海报/屏幕作为负样本或单独类别 | — | 按业务决定是否修 | 人工 |
| b.10 负样本不足 | 各类纯背景图 | — | 训练集占比约 10~20% | 人工确认无目标后入库 |
| b.11 标注不一致 | 不补新数据，先做旧标注一致性清洗 | — | — | 人工 |

## 3. 漏检类筛选及标注策略

| 原因 | 要什么 | 不要什么 | 说明（人工/自动） |
|---|---|---|---|
| b.4 远距离 | 干净背景的远距离目标样本 | 远距离 + 复杂背景（丢弃） | 高分辨率预标注 + 人工补漏；否则纯人工 |
| b.5 重叠 | 中低重叠、边界可辨样本，精确标注 | 重叠度高、人眼难分割的（不标，丢弃） | 人工判断 |
| b.6 姿态/阈值 | 蹲、弯腰、侧身等多场景分散收集 | 单一场景大量同姿态 | 人工判断；卡阈值先试阈值/后处理 |
| b.12 罕见姿态 | 躺/倒地样本，合规采集或专项补充 | — | 人工判断（训练集常为空白） |
| b.13 边缘截断 | 目标出入画面的连续帧 | — | 人工关键帧 + 视频帧传播 |
| b.14 成像退化 | 对应光照/天气真实样本 | 过度退化到人眼不可辨的 | 人工判断；不可辨转“场景无解”候选 |
| b.15 低对比 | 同类衣着/背景组合真实样本 | 人眼不可辨样本 | 人工判断 |
| b.16 分辨率/前处理 | 不补数据，改输入设定/多尺度 | — | 先诊断确认，再走输入方案 |

## 4. 生成 YOLO 格式数据标签并合并

新标注数据在合并前必须过一遍脚本检查，任何一条不过就打回标注环节：

- 格式检查：标签文件格式、类别 ID 合法、坐标归一化范围 [0,1]
- 完整性检查：每张 image 有对应 label（负样本允许空 label，但需在清单中显式声明）
- 越界检查：框坐标越界、零面积框、极端长宽比框
- 重复检查：与现有训练集做图像哈希/相似度去重
- 泄漏检查：同一段视频的帧不允许同时进训练集和测试集
- 规则符合性：再跑一遍标注规则质检（IoU 规则、抖动帧 100 张上限）

### badcase 数据的去向

新数据不能全进训练集。标准切法：

| 去向 | 比例建议 | 用途 |
|---|---|---|
| 训练集 | 70~80% | 让模型学 |
| badcase 验证集（固定保留） | 20~30% | 验证本次问题是否修复；永远不进训练 |
| 历史问题回归集 | 问题解决后归档代表性样本 | 防止未来迭代改坏 |

说明：

- 小样本时验证集可高于 20%
- 必须按视频/场景划分，禁止按帧随机划分
- 合并时半自动确认采样倍率、负样本占比、分布变化

### 版本登记（全自动）

每次合并生成一个数据集版本，manifest 记录：

```text
dataset_v2026.07.09_case123:
  基线: dataset_v2026.06.15
  新增: 480 张（来源 case #123, b.1 类 200 / b.4 类 280）
  修改: 36 张旧标注（b.1 溯源清洗，清单附后）
  删除: 12 张（重复/不合规）
  采样倍率: badcase x3
  划分方式: 按视频源，seed=42
  解决: 解决 xxx 客户误检问题
```

## 5. 训练参数：默认不调，用“配方模板”代替手调

### 基本原则

工具根据合并记录自动建议，人确认：

| 情况 | 训练方式 | 理由 |
|---|---|---|
| 只新增少量数据，没动旧标注 | 从上一版权重 finetune | 快，且大盘不易掉 |
| 做了旧标注清洗（b.1/b.11） | 建议全量重训 | 旧错误特征已进权重，finetune 洗不干净 |
| 新增大批数据或新场景（b.9/b.12） | 全量重训 | 分布变化大 |
| PTQ 无法达标需 QAT | QAT 模板训练 | 量化感知训练后重新导出 |

### 配方模板

- 模板 A「增量修复」: finetune, pretrain_model, names, epoch=100
- 模板 B「清洗重训」: from scratch/pretrain, 标准超参, 全量数据, epoch=300
- 模板 C「小目标专项」: 提高输入分辨率/多尺度（b.4）
- 模板 D「负样本强化」: 提高负样本采样权重（b.1/b.3/b.10）
- 模板 Q「QAT」: QAT 训练后重新导出 ONNX

人工节点：训练启动前确认“用哪个模板 + 是否有特殊改动”。不改动就确认；要改超参必须写理由并记录进版本。

## 6. 训练 Loss 监控：规则化告警 + 高 Loss 样本反查

自动监控规则（全自动，不需要人盯曲线）：

| 异常信号 | 自动判断 | 自动动作 |
|---|---|---|
| loss 出现 NaN/Inf | 学习率过大或脏标注 | 自动停训，通知人工，附最近 batch 数据清单 |
| loss 持续不降 | LR 不合适 / 数据问题 | 告警，继续训但标记 |
| loss 周期性突刺 | 某些 batch 含脏数据 | 记录突刺 iteration，反查新合入数据 |
| train loss 降、val 指标不升反降 | 过拟合 | 告警：建议降采样倍率或提前停 |
| val 指标早早平台期 | 正常收敛或欠拟合 | 报告中标记，不告警 |

训练完成自动生成训练报告：

- loss 曲线（分项：box/cls/obj）
- val 指标曲线
- 异常标记点
- 高 loss 样本清单
- 与上一版训练的曲线对比

人不需要盯训练过程，只看报告。

## 7. 测试评估：三层测试 + 事先定义的达标门禁

| 层 | 测什么 | 回答什么问题 |
|---|---|---|
| 1. badcase 验证集 | 本次客户问题 | 这次的问题修好了吗 |
| 2. 历史问题回归集 | 过往已解决的问题 | 有没有修一个坏一个 |
| 3. 专业测试集 | 大盘 Precision/Recall/mAP | 整体水平有没有掉 |

达标标准必须在训练前就定好（写进任务单，不能事后临时定）：

1. badcase 验证集：目标问题修复率 ≥ 80%（FP 消除率 / FN 召回率，按 case 定义）；关键客户指定 case 必须 100%
2. 历史回归集：通过率 100%（任何一条历史问题复发即不达标）
3. 专业测试集：mAP 下降 ≤ 0.5%，关键类别 Recall 下降 ≤ 1%

判定全自动，结论人工确认——特别是“修复率 80% 还差几个 case”的边缘情况，要人看剩下没修复的是不是“场景无解”候选。

## 8. 不达标回环

工具先自动判 A/B/C/D 并给证据，人工确认后按对应路径回去。D 类尤其要先查——很多“不达标”其实调个阈值就过了，重训是最贵的手段，应该最后用。

| 不达标类型 | 自动诊断信号 | 回到哪里 | 典型动作 |
|---|---|---|---|
| A. 本次问题没修好 | badcase 验证集指标没变化/变化小 | 数据环节 | 补数据；策略错则回归因；污染未清则继续溯源 |
| B. 修好了但大盘掉了 | 层 1 过、层 3 掉 | 合并配比环节 | 降采样/减新数据/改 finetune，重训，不用重新标注 |
| C. 历史问题复发 | 层 2 某条失败 | 规则一致性检查 | 清洗新旧标注规则冲突样本 |
| D. 只差临门一脚（卡阈值） | 失败样本置信度集中在阈值附近 | 后处理环节 | 调阈值/NMS，不重训 |

### 人工干预表单

系统给出：三层测试对比报告 + 不达标类型判定(A/B/C/D) + 失败样本清单

人工选择：

- 按建议路径返回（默认）
- 补充数据（填：类型 b.x、预估数量、来源）
- 调整配比/训练模板（选模板，填理由）
- 调整阈值/后处理（填参数）
- 部分场景判定无解 → 从达标口径中剔除并归档（填理由，进已知限制清单）
- 升级评审（拉算法负责人决策）

### 回环控制的三条要求

1. 每轮只改一类变量：这轮改了数据就不同时改超参，否则下轮结果无法归因——工具在任务单上强制记录“本轮改动项”
2. 轮数上限：同一 case 迭代 ≥3 轮仍不达标，自动升级到负责人评审，议题只有两个——继续投入还是判“场景无解/部分无解”
3. 每轮留档：数据版本、模板、改动项、三层测试结果全部入 case 记录

## 2. ONNX 基准校验

ONNX 不是导出来就能用，必须先和训练框架浮点结果对齐。

自动检查项：

| 检查项 | 说明 |
|---|---|
| ONNX 是否能加载 | 基本合法性 |
| 输入输出 shape 是否符合预期 | 防止动态维度/导出错误 |
| ONNXRuntime 推理是否正常 | 服务器侧验证 |
| ONNX 结果 vs 原训练框架结果 | 检查导出是否引入差异 |
| 检测框、置信度、类别是否一致 | 不能只看输出 tensor |

判断：

- ONNX 与训练框架浮点结果一致 → 进入平台转换
- ONNX 与训练框架浮点结果不一致 → 人工干预：ONNX 导出问题

### 人工干预点 1：ONNX 导出不一致

触发条件：

- ONNXRuntime 结果和 PyTorch/训练框架结果差异大
- 检测框数量不一致
- 置信度跨阈值
- 类别不一致
- 输出 shape 不符合预期

人工做什么：

- 检查 export 脚本
- 检查 opset
- 检查是否有不支持算子
- 检查前后处理是否被错误包含或遗漏
- 检查模型截断点是否正确

人工完成后：

```text
修正导出配置 → 重新导出 ONNX → 重新跑 ONNX 基准校验
```

## 4. 仿真结果与 ONNX 浮点结果如何比

所有平台统一使用一套比对逻辑。

比最终检测结果：

| 指标 | 说明 |
|---|---|
| 框数量是否一致 | 多框/少框 |
| 框 IoU | 同一目标框位置是否接近 |
| 置信度差异 | 是否跨过阈值 |
| 类别是否一致 | 是否分类错误 |
| FP/FN 是否新增 | 是否引入新误检/漏检 |

自动结论：

- 若最终检测一致，逐层误差可接受 → 平台转换/量化通过
- 若最终检测不一致，且逐层误差集中在量化层 → 量化问题
- 若最终检测不一致，但逐层误差不大 → 后处理/阈值/NMS/前处理配置问题
- 若转换后 shape/layout 异常 → 转换工具或算子问题

## 5. 板端验证怎么做

平台转换通过后，先做模型板端验证。验证包括效果测试和内存检查。

### 5.1 三类强制对比

1. 现有板端结果 vs 原板端结果
2. 现有板端结果 vs 基准结果
3. 现有板端结果 vs 仿真器结果

### 5.2 测试集

| 测试集 | 作用 |
|---|---|
| 本次 badcase 验证集 | 验证客户问题是否修复 |
| 历史问题回归集 | 验证过往问题是否复发 |
| 专业测试集 | 验证大盘指标是否稳定 |

### 5.3 达标标准（建议在进入转换前就写进任务单）

客户问题修复标准：

- 本次 badcase：FP 消除率 ≥ 80%，FN 召回率 ≥ 80%
- 关键客户指定 case 必须 100% 通过

历史回归标准：

- 历史问题集不得新增严重 FP/FN
- 关键历史 case 通过率 100%

板端性能标准：

- 帧率不低于要求
- 内存不超过限制
- 模型加载正常
- 长时间运行无异常

### 5.4 内存检查

| 检查项 | 目的 | 自动化程度 |
|---|---|---|
| 模型加载内存峰值 | 防止超出板端限制 | 高 |
| 推理过程中内存增长 | 检查泄漏 | 高 |
| 多次初始化/释放 | 检查资源释放 | 高 |
| 长时间循环推理 | 检查慢性泄漏 | 高 |
| 异常输入/空输入 | 检查异常路径释放 | 中 |

人工干预点：内存检查失败

触发条件：

- 内存泄漏
- 峰值超限
- 板端 crash
- 长稳测试失败

人工角色：软件/嵌入式负责人

人工处理：

- 查 malloc/free
- 查 buffer 生命周期
- 查模型加载/释放接口
- 查异常路径资源释放
- 查 NPU runtime 内存池

处理完成：

```text
修代码 → 重新编译 → 重新跑内存检查 → 重新跑板端回归
```

### 5.5 板端验证报告

- 板端驱动版本
- 代码、模型文件版本
- 测试集版本
- 新旧结果对比
- 与基准结果对比
- 与仿真结果对比
- 性能指标
- 是否通过门禁
- 未通过原因分类
- 建议下一步动作
