# 软件流程图评审与改版建议

**文档类型：** 流程评审  
**适用对象：** 软件 / 算法 / 项目负责人  
**版本：** v1.0  
**日期：** 2026-07-08

---

## 1. 总评

> **现有流程图更像“软件同事想象中的一条自动化直线”，而不是真实的算法问题闭环。**

最大问题不是“少了几个框”，而是**流程分层、分支、决策点、责任边界都没画对**。

按现有图推进的风险：
1. 平台做出来但算法同事觉得不好用
2. 关键节点仍靠人工补，只是“套了个平台壳子”
3. 软件/算法/IT 对职责理解不一致，项目卡住

---

## 2. 五大结构问题

### 2.1 把真实闭环画成“一条直线”

真实流程是**分诊 + 分叉 + 回环**，不是上传→分析→训练→导出→板测→release。

缺失的关键分支：
- 浮点正常/板端异常 → 量化/板端排查
- 浮点异常 → 模型/数据/标注策略
- 只改数据 vs 改量化 vs 改代码 vs 判定不修

### 2.2 “AI 自动判定”放得太靠前、太重

前期最值钱的不是“大模型替你判断”，而是：
- case 标准化
- 证据收集完整
- 浮点/板端结果统一拉齐

> **主引擎应该是规则化分诊，不是大模型推理。**

### 2.3 未拆分三套环境

| 环境 | 典型失败原因 |
|---|---|
| 训练服务器 | 数据/参数/脚本/资源 |
| Docker 转换 | ONNX 兼容/算子/PTQ/QAT/工具版本 |
| 板端 NPU | 量化掉点/前后处理/代码/驱动 |

三步不是同一环境、同一责任人、同一种失败模式，不能画成一条线。

### 2.4 缺少“输入/输出/责任人/状态”四要素

很多节点是“做一件事”而非“形成可交接产物”。例如“CV 多模态大模型自动判别”——输入数据结构？输出是标签还是任务单？谁审核？什么情况下回退？

### 2.5 人工干预画成“AI 失败兜底”而非“关键决策节点”

合理设计：**人工不是补锅角色，而是关键决策角色。**

---

## 3. 具体缺失清单

| 缺失项 | 说明 |
|---|---|
| Case 标准化入库 | 上传后直接进平台太粗 |
| 证据收集优先于归因 | 应先拿齐证据再分诊 |
| 浮点 vs 板端显式分叉 | 应作为一级分流点 |
| “无需训练修复”出口 | 配置/阈值/版本/不可复现/不纳入范围 |
| 训练前人审决策点 | 防止所有 case 自动扔去训练 |
| 训练后是否重新量化/编译判断 | 只改模型 vs 改代码 |
| 内部审批后再交付 | 不应直接 release 给客户 |

---

## 4. 最危险节点：自动标注直接进训练

原图“数据自动化标注（YOLOv8s）→ 通知模型训练”风险最大。

正确三步：
1. **预标注**（YOLOv8s 或自有模型）
2. **规则质检**（IoU 规则、数量约束、重叠过滤）
3. **人工抽检/修正**后才允许合入训练集

否则等于把“背景污染”的坑再踩一遍。

---

## 5. 改版原则

| 原则 | 说明 |
|---|---|
| 按阶段分图 | 问题分诊图 + 模型迭代交付图，不要一张图画全 |
| 按角色/系统画泳道 | 客户/平台/训练机/Docker/板端/人工决策 |
| 人工决策点是主节点 | 确认问题、是否训练、是否达标、是否发布 |
| 每步有明确输出物 | case ID、数据清单、浮点结果、板端结果、差异报告、任务单 |

---

## 6. 推荐新版流程 A（问题分诊闭环）

```
开始
  ↓
客户/项目接口人提交 badcase 数据
  ↓
Case 标准化入库（case ID、元数据、归档）
  ↓
数据预处理（抽帧、去重、数据清单）
  ↓
自动执行浮点推理
  ↓
自动执行板端推理
  ↓
自动结果对比与差异分析
  ↓
[判断1] 浮点与板端是否一致？
  ├─ 否 → 疑似量化/转换/板端问题 → 人工确认 → 排查任务单（软件方向）
  └─ 是 → [判断2] 是否满足客户预期？
            ├─ 否 → 疑似模型/数据问题 → 人工确认 → 训练/数据任务单（算法方向）
            └─ 是 → 疑似版本/配置问题 → 人工确认 → 核对任务单
  ↓
自动生成报告 → 归档历史 case 库 → 结束
```

---

## 7. 推荐新版流程 B（模型迭代交付，第二期）

```
训练/数据任务单
  ↓
数据筛选与标注策略确认 → 人工标注/预标注修正
  ↓
训练服务器启动训练 → 测试集评估
  ↓
[判断] 指标达标？
  ├─ 否 → 返回数据/训练策略调整
  └─ 是 → 模型导出
            ↓
          按目标平台分别：PTQ/QAT/转换/预编译
            ↓
          板端测试 → [判断] 板端达标？
            ├─ 否 → 量化/代码排查
            └─ 是 → 生成内部候选版本
                      ↓
                    自动生成 readme/doc/测试摘要
                      ↓
                    人工审批 → 进入客户同步流程
```

---

## 8. 结论

| 问题 | 结论 |
|---|---|
| 最大问题 | 把算法闭环画成 AI 自动流水线，弱化证据收集、分流、人工决策、环境边界 |
| 怎么改 | 重画成两张图：问题分诊闭环（先做）+ 模型迭代交付（后做） |
| 核心顺序 | **先证据、后分流、再决策** |
