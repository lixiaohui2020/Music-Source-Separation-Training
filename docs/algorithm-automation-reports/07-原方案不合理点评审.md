# 《算法自动化工具流程详尽方案》不合理点评审

**评审对象：** 算法自动化工具流程详尽方案.docx  
**版本：** v1.0  
**日期：** 2026-07-09  
**评审原则：** 对照真实开发流程（分诊 → 归因 → 标注 → 合并训练 → 三平台转换 → 板端验证 → 代码/内存/Gerrit → 内部候选交付）

---

## 0. 总评

原方案已经吸收了大量正确设计：

- 1 号通入库、版本对齐、浮点/板端两路推理
- FP/FN 归因表、筛选策略、数据合并检查
- 训练配方模板、Loss 监控、不达标回环
- V821 / V861 / V853 分平台转换
- 代码修改分支、内存检查、内部候选 release

**大方向正确，但还不能直接立项落地。**  
主要问题不是“缺几个框”，而是：

1. **章节结构把分诊、训练、部署混在一起**
2. **若干关键判定逻辑有歧义或错误**
3. **量化问题与模型问题的责任路由不清**
4. **标注自动化边界写反了几处**
5. **板端验证与交付顺序不完整**
6. **缺少 MVP 边界、状态机、人月与角色分工**

---

## 1. 结构不合理

### 1.1 “二、模型训练”标题过大，内容混杂

第二章标题是“模型训练”，但实际包含：

- 量化问题人工处理
- FP/FN 判定
- 误检/漏检归因
- 数据筛选与标注
- 数据合并
- 训练参数
- 测试评估
- 不达标回环

这会让软件/IT 同事误以为这些都属于“训练模块”。

**修改建议：** 拆成独立章节：

1. 问题分诊闭环  
2. 浮点问题归因与标注  
3. 数据合并与训练评估  
4. 平台转换与板端验证  
5. 代码合规与交付  

### 1.2 “部署代码编写”放在 ONNX/平台章节中间

原方案在平台转换后突然插入“6.部署代码编写”，再写板端验证。  
实际顺序应是：

```text
平台转换/量化/仿真通过
  ↓
板端验证（模型效果）
  ↓
判断是否涉及代码修改
  ↓
若涉及：代码整理 / 编译 / 内存检查 / 再板测
  ↓
交付
```

代码分支不应插在转换流程中间，而应作为板端验证后的独立阶段。

### 1.3 缺少“闭环分层”

原方案把 A（拿到数据到明确下一步）和 B（训练后到客户同步）揉成一篇长文，没有 MVP 边界。  
内部立项时容易做成大而全平台。

**修改建议：** 明确两期：

- 一期：分诊闭环 + 报告 + 历史 case  
- 二期：训练/转换/板测/交付自动化  

---

## 2. 分诊段不合理

### 2.1 量化问题只派给算法，责任路由过窄

原文：

> 浮点有框、板端没框（置信度差异大）→ 可能量化问题 → 派给算法

不合理。量化/板端不一致至少可能是：

- 量化精度问题
- 前后处理不一致
- NMS/阈值不一致
- 转换/layout 问题
- 板端 runtime/SDK 问题

**修改建议：**

```text
浮点正常、板端异常
  → 先自动做逐层量化诊断 + 前后处理对齐检查
  → 再派单：
     - 量化/转换问题 → 算法 + 平台工具链
     - 前后处理/代码问题 → 软件
     - 工具链 bug → V861 内部 / V853 芯原
```

### 2.2 缺少“前后处理对齐”前置条件

原方案直接“浮点结果 VS 板端结果”，没有先对齐：

- resize
- 归一化
- 颜色通道
- NMS
- 阈值

不对齐时，差异结论不可信。

### 2.3 缺少“是否复现客户问题”判断

原方案在“浮点和板端都错”后直接进入模型/数据问题。  
还应先确认：

> 浮点结果是否复现了客户反馈的问题？

若未复现，应走版本/配置/复现条件核对，而不是直接训练。

### 2.4 “浮点有框、板端没框”后的处理写成了代码编译

原文：

> 能解决，人工处理完，人工修改代码编译 → 给到自动化工具进行板端测试

这把“量化修复”和“代码修复”混为一谈。  
量化问题多数应回到：

- 重新 PTQ
- 混合量化
- QAT
- 调整截断点

只有确认是前后处理/runtime 时，才进代码编译。

## 3. FP/FN 判定表不合理

### 3.1 IoU 区间重叠，脚本无法执行

原文同时存在：

- 误检 FP：IoU 与所有真值框 `< th_iou`
- 定位/分类问题：IoU `0.1~0.5`

若 `th_iou=0.5`，则 IoU=0.3 同时命中两类。

**修改建议：互斥判定顺序**

```text
对每个检测框，取最大 IoU：
1) IoU ≥ 0.5 且类别对 → 正确
2) IoU ≥ 0.5 且类别错 → 分类错误
3) 0.1 ≤ IoU < 0.5 → 定位偏移
4) IoU < 0.1 → 误检 FP

对每个真值框：
5) 无 IoU ≥ 0.5 的检测框：
   - 有 0 < conf < 板端阈值 的匹配框 → 卡阈值型漏检
   - 完全无框 → 漏检 FN
6) 同一真值匹配多个检测框 → 重复框
```

### 3.2 缺少“假 FP / 基准漏标”检查

原方案已意识到“基准本身可能有问题”，但没有落成动作。  
应在人工二判时强制确认：

> 该位置是否真的没有目标？

若有目标，改判为“基准漏标”，修基准，不修模型。

### 3.3 定位/分类、重复框没有下游路径

判定表分出了 5 类，但后续只展开了误检/漏检。  
应补：

| 判定 | 去向 |
|---|---|
| 定位偏移 | 标注质量/回归困难路径 |
| 分类错误 | 类别标注与混淆负样本路径 |
| 重复框 | NMS/后处理路径，不进数据训练 |

## 4. 归因与筛选策略不合理

### 4.1 漏检原因缺 b.16（最高优先级漏项）

原方案漏检表到 b.15，缺少：

> **b.16 输入分辨率/前处理损失**

表现与 b.4 远距离漏检很像，但修法完全不同：

- b.4：补干净背景远距数据
- b.16：改输入设定/多尺度，不补数据

若不先排除 b.16，会把分辨率问题误当成远距数据问题，白跑一轮训练。

### 4.2 b.4 写成“自动标注”逻辑矛盾

远距离漏检之所以漏，是因为模型检不出。  
预标注模型在这些样本上通常也检不出，不能写“自动标注，人工判断”。

**修改建议：**

- 优先：服务器高分辨率/切图预标注 + 人工补漏
- 否则：纯人工标注

### 4.3 b.3 条状物负样本写成“自动标注”不合适

负样本没有框可标，关键动作是：

> 人工确认图中真的没有人

应改为：

> 自动收集，人工确认无目标后入库

### 4.4 b.15 有原因、无筛选策略

漏检原因有 b.15，筛选策略表没有对应行，应补齐。

### 4.5 大模型匹配缺少约束

原方案写“大模型反馈信息与如下进行匹配”，方向对，但缺约束：

1. 输出只能是 b.1~b.16 或“无法归类”
2. 必须附诊断信号数值
3. 支持多标签
4. 人工改判结果回写，用于持续修正

## 5. 数据合并与训练不合理

### 5.1 badcase 验证集只留 10% 偏少

原文训练集 90%、验证集 10%。  
若 badcase 只有几十到一两百张，10% 验证集过小，修复结论不稳定。

**修改建议：**

- 小样本：验证集至少保留 20~30%，或按视频源固定留出
- 必须按视频/场景划分，禁止按帧随机划分

### 5.2 达标门禁漏了专业测试集

原文达标标准只写了：

1. badcase 验证集 ≥ 80%
2. 历史回归集 100%

但三层测试里有专业测试集，门禁却没写。  
应补：

```text
③ 专业测试集：mAP / Recall 下降不超过允许阈值
```

### 5.3 训练模板偏少

原文只有：

- 模板 A 增量修复
- 模板 B 清洗重训

建议至少再加：

- 模板 C：小目标专项
- 模板 D：负样本强化

否则 b.4 / b.1 / b.10 场景缺少默认配方。

### 5.4 合并配比缺少“采样倍率”落地项

原文提到过拟合风险，但合并段没有明确：

- badcase 采样倍率
- 负样本占比上限
- 分布变化报告

这些应作为合并时的半自动确认项。

## 6. 平台转换与板端验证不合理

### 6.1 板端验证位置偏后且不完整

原方案在代码分支后才详细写板端验证。  
实际上：

1. 平台转换后先做**模型板端验证**
2. 若改代码，再做**代码变更后的板端回归 + 内存检查**

两段板测不能合并成一次。

### 6.2 三类对比没有成为强制门禁

原方案提到新旧对比、基准对比、仿真对比，但没有明确为板端通过的强制条件。

应强制：

1. 现有板端 vs 原板端  
2. 现有板端 vs 基准  
3. 现有板端 vs 仿真器  

### 6.3 V853 外部等待状态写了，但总状态机缺失

原文正确提出 `external_waiting`，但全文没有统一任务状态机。  
没有状态机，人工干预后无法稳定回到自动化。

### 6.4 V821 QAT 回到训练后，缺少“QAT 专用训练模板”

原文写“需要 QAT → 回到训练环节”，但训练模板只有 A/B，没有 QAT 模板。

---

## 7. 交付段不合理

### 7.1 Gerrit 与审批顺序略乱

原文：

```text
生成内部候选包 → 自动生成文档 → 人工审批 → 提交/合入 Gerrit
```

更合理的是：

```text
准备提交材料
  ↓
Gerrit Review / 合入
  ↓
生成内部候选 release 包
  ↓
算法+软件负责人审批
  ↓
归档
  ↓
进入客户同步
```

或者明确：

- 代码变更走 Gerrit
- 仅模型替换可不走代码合入，但仍需审批

### 7.2 缺少客户反馈回环

交付到“归档版本与测试报告”就结束，没有：

> 客户确认解决 / 未解决 → 回写 case 库

闭环没有真正闭上。

### 7.3 缺少“只发单平台 / 多平台”策略

你们同时有 V821 / V861 / V853。  
交付时应支持：

- 只发布问题平台
- 或三平台同步发布

原方案未定义。

---

## 8. 工程落地缺失

原方案作为“详尽方案”仍缺以下立项必需内容：

| 缺失项 | 影响 |
|---|---|
| MVP 范围与非目标 | 容易做成大平台 |
| 统一任务状态机 | 人工进出自动化无机制 |
| 角色 RACI | IT/软件/算法职责不清 |
| 人月估算 | 无法立项 |
| 验收指标 | 无法判断一期是否成功 |
| 规则库版本管理 | 标注策略无法沉淀 |
| 环境适配器定义 | 训练机/Docker/板端接口不清 |

## 9. 必须修改的高优先级清单

| 优先级 | 问题 | 修改动作 |
|:---:|---|---|
| P0 | IoU 判定区间重叠 | 改为互斥判定顺序 |
| P0 | 缺少 b.16 | 补入漏检归因，并在 b.4 前先排除 |
| P0 | b.4 自动标注矛盾 | 改为高分辨率预标注或纯人工 |
| P0 | 量化问题只派算法 | 改为量化诊断后按原因派算法/软件/工具链 |
| P0 | 缺少前后处理对齐 | 比对前强制对齐 |
| P1 | 假 FP / 基准漏标 | 人工二判增加确认项 |
| P1 | 章节结构混杂 | 按闭环重排 |
| P1 | 板测与代码分支顺序 | 先模型板测，再代码分支 |
| P1 | 专业测试集门禁缺失 | 补进达标标准 |
| P1 | 负样本“自动标注” | 改为自动收集 + 人工确认无目标 |
| P2 | 验证集 10% 过少 | 小样本提高到 20~30% |
| P2 | 缺状态机/MVP/人月 | 修订版方案补齐 |
| P2 | 客户反馈未回写 | 交付后闭环回写 case |

---

## 10. 结论

原方案**可作为内部讨论底稿，不能直接作为建设蓝图**。

建议处理方式：

1. 保留其正确骨架与表格资产  
2. 按本评审修正 P0/P1 问题  
3. 重排为“分诊闭环 + 训练交付闭环”两层方案  
4. 补齐状态机、人工决策表单、MVP 与人月
