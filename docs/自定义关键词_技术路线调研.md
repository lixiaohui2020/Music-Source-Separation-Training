## 1. 总览

主流自定义关键词技术路线可分为 **8 类**。在低功耗量产场景里，真正常用的是前 **5** 类；后几类偏研究或高端个性化。

| # | 技术路线 | 自定义方式 | 是否重训整网 | 典型延迟 | 低功耗适配 |
|---|---|---|---|---|---|
| 1 | 闭集判别式重训 | 云端按词重训分类器 | 是 | 天～周 | ⭐⭐⭐ 最好（固定小模型） |
| 2 | TTS 合成训练 | 文字→TTS→训练 | 是 | 小时～天 | ⭐⭐⭐ 好 |
| 3 | 迁移/微调/蒸馏 | 大模型→小模型适配新词 | 部分 | 小时～天 | ⭐⭐⭐ 好 |
| 4 | Query-by-Example 少样本注册 | 用户说几遍建模板 | 否 | 秒～分钟 | ⭐⭐ 中（常驻编码器） |
| 5 | 文本注册 / Zero-shot | 输入文字直接出词 | 否 | 秒 | ⭐⭐ 中 |
| 6 | 音素/混合 ASR 检出 | 音素解码+关键词匹配 | 否/轻量 | 秒 | ⭐ 中～差（算力更大） |
| 7 | 开放词表 + 声纹多任务 | 词+说话人同时约束 | 否 | 秒 | ⭐⭐ 中 |
| 8 | 端侧自学习/增量适应 | 部署后伪标签微调 | 轻量微调 | 持续 | ⭐ 难（要算力/存储） |

**行业共性：**

- 硬件（网络结构）通常固定，**换的是权重文件 + 阈值**
- 量产自定义分两档：**精训（真人语料）** vs **快训（TTS/Console）**
- 很少让客户在端侧从零训练；端侧“说几遍学会”多是 **enrollment**，不是完整重训

## 2. 路线 1：闭集判别式重训（Closed-set Retrain）

### 2.1 原理

固定网络结构（FSMN / DS-CNN / CNN-RNN 等），将每个客户唤醒词（或词组）作为**独立分类类别**，用大量正负样本重新训练，导出 INT8 量化权重，通过 Flash/OTA 下发。

```
客户提交词条 + 语料
        ↓
云端训练（固定 backbone + 新分类头）
        ↓
量化 → 校准阈值 → 交付模型文件
        ↓
设备 OTA / 产线烧录
```

### 2.2 特点

| 优点 | 缺点 |
|---|---|
| 量产 FAR/FRR 最好控 | 换词需重新训练/下发 |
| 端侧推理路径固定，功耗可预测 | 语料成本高（常需上万条） |
| 与 always-on 小 SRAM 方案最匹配 | 网络结构一般不开放给客户改 |

### 2.3 代表厂商/方案

| 厂商 | 做法 |
|---|---|
| **乐鑫** | WakeNet 正式定制：客户语料 ≥2 万条，或厂商代采，约 2–3 周出模 |
| **CEVA** | WhisPro trigger phrase customization service |
| **Sensory** | VoiceHub 产线级品牌词训练 |
| **国内芯片原厂** | 普遍采用“付费出模”模式 |

### 2.4 参考链接

- 乐鑫定制流程：https://docs.espressif.com/projects/esp-sr/en/latest/esp32s3/wake_word_engine/ESP_Wake_Words_Customization.html
- CEVA WhisPro 发布说明：https://www.ceva-ip.com/press/ceva-introduces-whispro-neural-network-based-speech-recognition-technology-for-voice-assistants-and-iot-devices/

## 3. 路线 2：TTS 合成训练（TTS-only / TTS-augmented）

### 3.1 原理

客户只提供**文字词条** → 用 TTS 合成多说话人、多韵律的正样本 → 叠加噪声/混响/设备频响增强 → 训练 KWS 分类器。

可纯 TTS，也可 **TTS + 少量真人** 混合。

### 3.2 特点

| 优点 | 缺点 |
|---|---|
| 出模快、成本低 | 精度通常略低于真人精训 |
| 适合 POC / 中小客户快速试词 | 对 TTS 与目标设备声学差异敏感 |
| 语料规模可快速扩展 | 近音词、口音仍需额外验证 |

### 3.3 代表方案

| 方案 | 说明 |
|---|---|
| **乐鑫 TTS Pipeline V2/V3** | 宣称可达真人语料模型约 90–98% 效果；支持中/英/日/法等 |
| **多家原厂内部快通道** | 不公开细节，模式类似 |
| **学术研究** | pseudo-TTS 做 few-shot 数据扩增 |

### 3.4 参考链接

- 乐鑫 TTS 训练说明（GitHub Issue #88）：https://github.com/espressif/esp-sr/issues/88
- TTS/pseudo-TTS 做 FS-KWS 论文：https://export.arxiv.org/pdf/2210.02732v2.pdf
- Interspeech 2024 自定义唤醒（含 TTS 表征思路）：https://www.isca-archive.org/interspeech_2024/v24_interspeech.pdf

## 4. 路线 3：迁移学习 / 微调 / 知识蒸馏（Transfer / Distill）

### 4.1 原理

- 用**大模型**（通用词表 KWS 或 SSL 模型如 wav2vec2）作教师
- **蒸馏**到端侧小网络（DS-CNN / ResNet / 小 FSMN）
- 对新词做小数据微调，或只训练最后分类头

```
Teacher（大模型 / SSL）
        ↓ 蒸馏
Student（端侧小网络，10K～1M 参数）
        ↓ 少量新词数据微调
客户定制模型
```

### 4.2 特点

| 优点 | 缺点 |
|---|---|
| 比从零训练省数据 | 仍需 ML 流水线，通常由厂商云端完成 |
| 端侧仍是小分类器，功耗友好 | 蒸馏质量依赖教师与域匹配 |
| 可兼顾多词表泛化与单客户定制 | 客户侧难以自助完成 |

### 4.3 代表方案

- **ADI 团队方案**：wav2vec2 教师 → 边缘学生模型，用户少样本即可加词
- **各类“通用 KWS backbone + 客户头”** 方案

### 4.4 参考链接

- ADI Custom KWS at the Edge：https://developer.analog.com/newsroom/custom-keyword-spotting-at-the-edge-lightweight-accurate-and-easy-to-use
- Hello Edge（小模型架构基线）：https://arxiv.org/abs/1711.07128

## 5. 路线 4：Query-by-Example / 少样本注册（QbyE / Few-shot Enrollment）

### 5.1 原理

离线训练好**音频编码器**（embedding 网络）；自定义时用户说 3–10 遍，计算**原型向量（prototype）**；推理时对滑动窗音频算 embedding，与原型比较余弦距离/欧氏距离，超过阈值即触发。

```
离线：在大规模词表上训练 Encoder（Prototypical / Triplet / GE2E）
        ↓
注册：用户说 N 遍 → 计算 prototype
        ↓
推理：live audio → embedding → distance(prototype) > threshold → 命中
```

### 5.2 特点

| 优点 | 缺点 |
|---|---|
| **不重训整网**，换词极快（秒～分钟） | 对噪声、口音、远场更敏感 |
| 适合“终端用户自定义” | 需要 open-set 拒绝机制（dummy prototype / 阈值） |
| Always-on 只需常驻编码器 | 编码器比纯分类器略重 |

### 5.3 关键技术

- **Prototypical Networks**：每类用 support set 均值作原型
- **Triplet Loss**：拉近同类、推远异类
- **Open-set**：额外 unknown 类原型，拒绝非关键词
- **GE2E**：广义端到端，enrollment centroid 与 test embedding 比对

### 5.4 代表方案

| 方案 | 说明 |
|---|---|
| **Sensory Personalized Wake Word** | 用户说约 4 遍，<15–20 秒，端侧完成注册 |
| **Qualcomm SVA** | 支持 OEM 词 + 用户自定义短语（产品层） |
| **学术研究** | Prototypical Networks、Triplet、GE2E-KWS |

### 5.5 参考链接

- Sensory Personalized Wake Word：https://sensory.com/product/personalized-wake-word/
- Few-shot Open-set On-device KWS（Interspeech 2023）：https://www.isca-archive.org/interspeech_2023/rusci23_interspeech.pdf
- Few-Shot KWS with Prototypical Networks：https://dl.acm.org/doi/fullHtml/10.1145/3529399.3529443

## 6. 路线 5：文本注册 / Zero-shot（Text Enrollment）

### 6.1 原理

客户输入**文字**即可自定义，两条子路径：

- **A. TTS 桥接**：文字 → TTS 合成音频 → 再走 QbyE 注册（embedding 原型）
- **B. 跨模态匹配**：文本/音素与语音投到同一 embedding 空间，直接匹配（audio-text cross-modal）

### 6.2 特点

| 优点 | 缺点 |
|---|---|
| 体验最好：“打字即出词” | 近音混淆、多语言发音差异是难点 |
| 无需客户录音 | 跨模态对齐质量决定上限 |
| 秒级出模 | 量产 FAR 控制通常不如精训闭集 |

### 6.3 代表方案

| 方案 | 说明 |
|---|---|
| **Picovoice Porcupine Console** | 浏览器输入短语 → 数秒出模型 → 下载平台优化文件 |
| **GE2E-KWS** | 支持 audio 或 text 注册，text 经 TTS 转 embedding |
| **跨模态 audio-text matching** | 学术论文方向 |

### 6.4 参考链接

- Picovoice Porcupine：https://picovoice.ai/products/voice/wake-word/
- Picovoice 完整技术指南：https://picovoice.ai/blog/complete-guide-to-wake-word/
- GE2E-KWS（zero-shot）：https://arxiv.org/html/2410.16647v1
- Custom Wake Word Detection（Interspeech 2024）：https://www.isca-archive.org/interspeech_2024/v24_interspeech.pdf

## 7. 路线 6：音素 / 混合 ASR 关键词检出（Phoneme / Hybrid ASR）

### 7.1 原理

用通用**音素识别器**或小型 ASR，在音素/字格上做关键词搜索（phone lattice / WFST / filler model）。换词 = 改词典/语法，**不换声学模型**。

### 7.2 特点

| 优点 | 缺点 |
|---|---|
| 开放词表能力强 | 算力和内存通常高于专用 KWS |
| 换词成本低（改语法） | always-on 功耗压力大 |
| 适合长命令词 | 短唤醒词 FAR 控制往往不如专用判别模型 |

### 7.3 代表

- 传统 DTW / 音素 HMM-DNN 方案（早期自定义）
- 部分车载/远场方案：“小 ASR + 关键词语法”

### 7.4 参考链接

- GE2E-KWS 文中对 DTW/音素路线的对比讨论：https://arxiv.org/html/2410.16647v1

## 8. 路线 7：开放词表 + 声纹多任务（Personalized OV-KWS）

### 8.1 原理

同时做**关键词匹配**与**说话人验证（SV）**：不但要说对词，还要是注册用户，降低他人误触发。

```
输入音频
   ├─ KWS 分支：是否为目标词？
   └─ SV 分支：是否为目标用户？
         ↓
   两者同时满足 → 唤醒
```

### 8.2 特点

| 优点 | 缺点 |
|---|---|
| 适合可穿戴、户外、个人设备 | 比纯 QbyE 多一路 SV 计算 |
| 显著降低他人误触发 | 需要用户声纹注册 |
| Sensory 等产品把 biometric wake word 作为可选层 | 多任务 loss 权重需调优 |

### 8.3 参考链接

- PCOV-KWS 论文：https://arxiv.org/pdf/2603.18023
- Sensory Personalized（含 voiceprint 可选）：https://sensory.com/product/personalized-wake-word/

## 9. 路线 8：端侧自学习 / 增量适应（On-device Self-learning）

### 9.1 原理

部署后用**伪标签** / 距离准则收集新环境数据，增量微调编码器或原型，适应麦克风与噪声域变化。

### 9.2 特点

| 优点 | 缺点 |
|---|---|
| 提升长期个性化与域适应 | 对 MCU 算力、存储要求高 |
| 无需回传云端 | 伪标签质量难保证 |
| 适合固定安装设备长期运行 | 量产案例较少 |

### 9.3 参考链接

- Self-Learning Personalized KWS on Ultra-Low-Power Audio Sensors：https://arxiv.org/html/2408.12481v1

## 10. 与客户类型的对应关系

| 客户诉求 | 主流技术路线 | 典型交付方 |
|---|---|---|
| OEM 品牌词，要量产级 FAR/FRR | **路线 1**（常叠加路线 2 做数据增强） | 芯片/算法原厂代训 |
| 快速试词、低成本验证 | **路线 2** 或 **路线 5** | TTS 管道 / Console |
| 终端用户自己设唤醒语 | **路线 4**（可加路线 7） | 端侧 enrollment SDK |
| 打字出词、多平台软件授权 | **路线 5** | Picovoice 类中间件 |
| 极大开放词表 / 长命令 | **路线 6**（或 5+4） | 偏重方案，少用于 μW always-on |
| 户外/个人设备防误触 | **路线 7** | Sensory 等 biometric 方案 |
| 长期域适应 | **路线 8** | 研究/高端 IoT |

## 11. 对自研 Audio Core / FSMN 方案的映射建议

> 假设现状：FSMN ~750K 参数；SRAM 预算约 100KB always-on 域；已有误检/漏检问题

### 11.1 分场景推荐路线

| 场景 | 推荐路线 | 说明 |
|---|---|---|
| **OEM 品牌自定义（优先做）** | 路线 1 + 路线 2 | 精训走真人/代采语料；快训走 TTS；交付量化 FSMN 权重 + 阈值 |
| **用户自定义（中期）** | 路线 4 | 冻结编码器 + 少样本原型；不必每次重训 750K |
| **打字即用（体验型）** | 路线 5 | 文本→TTS→注册；或接第三方 Console |
| **100KB SRAM 约束** | 路线 4（Stage-1）+ 路线 1（Stage-2） | always-on 小编码器/小分类器；命中后加载 750K FSMN 复检 |

### 11.2 不建议的做法

- ❌ 直接把 750K FSMN 换成 30KB 单级小模型（效果大概率更差）
- ❌ 在 100KB SRAM 里常驻 750K 权重（约 750KB INT8，放不下）
- ❌ 指望“换小模型”解决现有误检/漏检（应先查数据、阈值、级联）

### 11.3 推荐架构（级联）

```
Always-on 域（≤100KB SRAM）
  Stage-1：小模型 / 编码器（~25K–40K 参数，高召回）
       ↓ 可疑命中
  Stage-2：从 Flash 加载 FSMN 750K 复检（压误检）
       ↓ 确认
  上电 NPU / 主系统
```

## 13. 总结

| 问题 | 结论 |
|---|---|
| **量产自定义主流怎么做？** | **闭集重训（路线 1）± TTS 增强（路线 2）**；原厂代训 + OTA 下发权重 |
| **体验型自定义主流怎么做？** | **Embedding 少样本注册（路线 4）** 或 **文本注册（路线 5）** |
| **硬件要不要改？** | 通常**不改**；换权重 + 阈值；少数方案需常驻编码器 |
| **低功耗 always-on 首选？** | 路线 1/2/3（小固定分类器）；路线 4/5 需评估编码器常驻成本 |
| **不推荐用于 μW 唤醒？** | 路线 6（音素 ASR）、路线 8（端侧自学习）作为第一选择 |
| **100KB SRAM + 750K FSMN？** | **级联**：Stage-1 小模型常驻 + Stage-2 FSMN 复检，不要单级硬换 |
